📐 RESTful API Design Principles

Resource Naming Conventions

# ✅ GOOD - Nouns, plural, lowercase GET /users # List all users GET /users/{id} # Get specific user POST /users # Create new user PUT /users/{id} # Update entire user PATCH /users/{id} # Partial update DELETE /users/{id} # Delete user # Nested resources GET /users/{id}/orders # User's orders POST /users/{id}/orders # Create order for user GET /orders/{orderId}/items # Order items # ❌ BAD - Verbs, inconsistent GET /getUsers # Don't use verbs POST /createUser # HTTP method indicates action GET /user # Use plural GET /Users # Use lowercase

🔄 API Versioning Strategies

1. URL Path Versioning (Most Common)

GET /api/v1/users GET /api/v2/users # Pros: Clear, easy to implement, cacheable # Cons: URL changes between versions

2. Header Versioning

GET /api/users Accept: application/vnd.myapi.v2+json # Pros: Clean URLs # Cons: Less discoverable, harder to test in browser

3. Query Parameter

GET /api/users?version=2 # Pros: Easy to test # Cons: Can be confused with filters
💡 Recommendation: Use URL path versioning for public APIs. It's the most explicit and widely adopted approach. Start with v1 and only introduce v2 when you have breaking changes.

Version Lifecycle Management

# Version deprecation cycle v1 Released: Jan 2023 v2 Released: Jan 2024 ← v1 deprecated, 6-month sunset v1 Sunset: Jul 2024 ← v1 no longer available # Deprecation headers Deprecation: true Sunset: Sat, 01 Jul 2024 00:00:00 GMT Link: </api/v2/users>; rel="successor-version"

JWT Security

// ✅ DO - Use strong secret (256+ bits) or RS256 - Short expiration (15-60 min for access token) - Use refresh tokens - Validate all claims - Store in httpOnly cookies (not localStorage) - Include jti (JWT ID) for revocation // ❌ DON'T - Store sensitive data in payload - Accept tokens from URL params - Use none algorithm - Ignore expiration - Use same secret across environments

Rate Limiting

💾 Database Design Best Practices

Indexing Strategy

# Rules for indexes 1. Index all WHERE clause columns 2. Index JOIN columns 3. Index ORDER BY columns 4. Use composite indexes for multi-column queries 5. Consider index order (leftmost prefix rule) # MongoDB example db.users.createIndex({ email: 1 }, { unique: true }) db.orders.createIndex({ userId: 1, createdAt: -1 }) # SQL example CREATE INDEX idx_user_email ON users(email); CREATE INDEX idx_orders_user_date ON orders(user_id, created_at DESC);

Query Optimization Checklist

  • Avoid SELECT * - select only needed columns
  • Use EXPLAIN/EXPLAIN ANALYZE for query analysis
  • Implement pagination for large result sets
  • Use connection pooling
  • Avoid N+1 queries with proper joins/eager loading
  • Use database-level constraints
  • Denormalize for read-heavy workloads
  • Archive old data to maintain table size

Cache Patterns

CACHE ASIDE (Lazy Loading) ┌─────────┐ 1. Request ┌─────────┐ │ Client │ ──────────────────>│ App │ └─────────┘ └────┬────┘ │ 2. Check Cache ▼ ┌─────────┐ │ Cache │ ← 3a. Hit: Return └────┬────┘ │ 3b. Miss ▼ ┌─────────┐ │ DB │ ← 4. Query DB, Update Cache └─────────┘ WRITE THROUGH ┌─────────┐ Write ┌─────────┐ │ Client │ ──────────────>│ App │ └─────────┘ └────┬────┘ │ ┌──────────────┼──────────────┐ ▼ ▼ ▼ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ Cache │ │ DB │ │ Both │ └─────────┘ └─────────┘ │ Updated │ └─────────┘ WRITE BEHIND (Write Back) App writes to Cache → Background sync to DB (async)

Redis Implementation

const Redis = require('ioredis'); const redis = new Redis(); // Cache-aside pattern async function getUser(id) { // 1. Check cache const cached = await redis.get(`user:${id}`); if (cached) return JSON.parse(cached); // 2. Query database const user = await db.findOne({ _id: id }); // 3. Set cache with TTL await redis.setex( `user:${id}`, 3600, // 1 hour TTL JSON.stringify(user) ); return user; } // Cache invalidation async function updateUser(id, data) { const user = await db.updateOne({ _id: id }, data); await redis.del(`user:${id}`); // Invalidate cache return user; }

What to Cache

✅ Cache These:
  • User sessions and profiles
  • Configuration/settings
  • Expensive query results
  • API rate limit counters
  • Precomputed aggregations
⚠️ Don't Cache These:
  • Sensitive data (without encryption)
  • Real-time data (stock prices, etc.)
  • Large objects (use CDN instead)
  • Data that changes frequently

⚠️ Error Handling Patterns

Structured Error Handling

// Custom error classes class AppError extends Error { constructor(message, statusCode, code) { super(message); this.statusCode = statusCode; this.code = code; this.isOperational = true; Error.captureStackTrace(this, this.constructor); } } class ValidationError extends AppError { constructor(message, details = []) { super(message, 422, 'VALIDATION_ERROR'); this.details = details; } } class NotFoundError extends AppError { constructor(resource) { super(`${resource} not found`, 404, 'NOT_FOUND'); } } class ConflictError extends AppError { constructor(message) { super(message, 409, 'CONFLICT'); } }

Global Error Handler

// Error handling middleware const errorHandler = (err, req, res, next) => { err.statusCode = err.statusCode || 500; err.code = err.code || 'INTERNAL_ERROR'; // Log error logger.error({ message: err.message, stack: err.stack, requestId: req.id, path: req.path, method: req.method, user: req.user?.id }); // Operational vs Programming errors if (!err.isOperational) { // Programming error - don't leak details return res.status(500).json({ success: false, error: { code: 'INTERNAL_ERROR', message: 'Something went wrong' } }); } // Send response res.status(err.statusCode).json({ success: false, error: { code: err.code, message: err.message, ...(err.details && { details: err.details }) }, ...(process.env.NODE_ENV === 'development' && { stack: err.stack }) }); };

Async Handler Wrapper

// Wrapper for async route handlers const asyncHandler = fn => (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next); // Usage app.get('/users/:id', asyncHandler(async (req, res) => { const user = await User.findById(req.params.id); if (!user) throw new NotFoundError('User'); res.json({ success: true, data: user }); }));

📨 Background Jobs & Queues

When to Use Background Jobs

  • Email sending (notifications, newsletters)
  • File processing (image resizing, PDF generation)
  • Third-party API calls (webhooks, integrations)
  • Report generation (CSV exports, analytics)
  • Scheduled tasks (daily cleanup, aggregations)
  • Heavy computations (data processing, ML inference)

Bull Queue (Redis-based)

const Queue = require('bull'); // Create queue const emailQueue = new Queue('email sending', { redis: { host: 'localhost', port: 6379 } }); // Add job await emailQueue.add('send-welcome', { to: user.email, template: 'welcome', data: { name: user.name } }, { attempts: 3, backoff: { type: 'exponential', delay: 1000 }, removeOnComplete: 100, removeOnFail: 50 }); // Process job emailQueue.process('send-welcome', async (job) => { const { to, template, data } = job.data; await sendEmail(to, template, data); }); // Error handling emailQueue.on('failed', (job, err) => { logger.error('Job failed', { jobId: job.id, error: err.message }); });

🚩 Feature Flags & Rollout

Feature Flag Implementation

// Feature flag service class FeatureFlags { constructor() { this.flags = { 'new-dashboard': { enabled: true, rolloutPercentage: 50, whitelist: ['user_123'], blacklist: [] }, 'beta-feature': { enabled: false, whitelist: ['admin'] // Only for admins } }; } isEnabled(flagName, user) { const flag = this.flags[flagName]; if (!flag || !flag.enabled) return false; // Whitelist check if (flag.whitelist?.length) { return flag.whitelist.includes(user.id) || flag.whitelist.includes(user.role); } // Blacklist check if (flag.blacklist?.includes(user.id)) { return false; } // Percentage rollout if (flag.rolloutPercentage) { return this.hashUserId(user.id) < flag.rolloutPercentage; } return true; } hashUserId(id) { // Consistent hash for same user return parseInt(crypto.createHash('md5') .update(id) .digest('hex') .substring(0, 2), 16) % 100; } } // Middleware const featureFlag = (flagName) => (req, res, next) => { if (!flags.isEnabled(flagName, req.user)) { return res.status(404).json({ error: 'Not found' }); } next(); }; // Usage app.get('/new-dashboard', featureFlag('new-dashboard'), handler);

❌ Common Anti-Patterns to Avoid

🚫 Architecture Anti-Patterns:
  • Monolithic database for microservices
  • Shared state between services
  • Synchronous calls for everything
  • No circuit breakers for external calls
  • Hardcoded configuration values
🚫 Security Anti-Patterns:
  • Storing passwords in plain text
  • Using JWT in localStorage (use httpOnly cookies)
  • Trusting client-side validation only
  • Exposing internal errors to clients
  • No rate limiting on auth endpoints
🚫 Performance Anti-Patterns:
  • SELECT * from database
  • No pagination on list endpoints
  • N+1 query problem
  • Synchronous heavy operations in request
  • No caching strategy
🚫 Code Anti-Patterns:
  • Catching and ignoring exceptions
  • Global mutable state
  • God classes/functions
  • Copy-paste code instead of abstraction
  • No input validation

Scaling Decision Matrix

// When to scale horizontally: - Stateless services (API servers) - Read-heavy workloads - Need for fault tolerance - Unpredictable traffic patterns // When to scale vertically: - Stateful services (databases) - Write-heavy workloads - Complex queries - Strong consistency requirements // When to use both: - Database: Primary (vertical) + Read replicas (horizontal) - API: Auto-scaling group (horizontal) with adequate resources (vertical)
