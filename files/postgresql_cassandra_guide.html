Data Distribution

🐘 PostgreSQL:
  • Single database server (primary)
  • Replicas for read scaling
  • Sharding requires external tools (Citus)
  • Data stored in tables with rows/columns
🌍 Cassandra:
  • Ring topology with token ranges
  • Partition key determines node placement
  • Automatic data distribution
  • Replication factor controls copies

🐘 PostgreSQL CRUD Operations

Insert

-- Basic insert INSERT INTO users (email, name, age) VALUES ('john@example.com', 'John Doe', 30); -- Multiple rows INSERT INTO users (email, name) VALUES ('jane@example.com', 'Jane'), ('bob@example.com', 'Bob'); -- Insert with RETURNING INSERT INTO users (email, name) VALUES ('new@example.com', 'New User') RETURNING *; -- Insert from select INSERT INTO users_backup SELECT * FROM users WHERE status = 'inactive';

Select

-- Basic select SELECT * FROM users; -- Select with conditions SELECT name, email FROM users WHERE age > 25 AND status = 'active'; -- Order, limit, offset SELECT * FROM users ORDER BY created_at DESC LIMIT 10 OFFSET 20; -- Distinct SELECT DISTINCT status FROM users; -- Aggregations SELECT status, COUNT(*) as count, AVG(age) as avg_age, MAX(age) as max_age FROM users GROUP BY status HAVING COUNT(*) > 5;

Update

-- Basic update UPDATE users SET status = 'inactive', updated_at = NOW() WHERE id = 1; -- Update with returning UPDATE users SET age = age + 1 WHERE id = 1 RETURNING *; -- Update from another table UPDATE orders o SET status = u.status FROM users u WHERE o.user_id = u.id; -- Upsert (INSERT ... ON CONFLICT) INSERT INTO users (email, name) VALUES ('john@example.com', 'John Updated') ON CONFLICT (email) DO UPDATE SET name = EXCLUDED.name;

Delete

-- Basic delete DELETE FROM users WHERE id = 1; -- Delete with returning DELETE FROM users WHERE status = 'inactive' RETURNING *; -- Truncate (faster, resets sequences) TRUNCATE TABLE users RESTART IDENTITY;

🐘 PostgreSQL Advanced Queries

JOINs

-- INNER JOIN SELECT u.name, o.total, o.status FROM users u INNER JOIN orders o ON u.id = o.user_id; -- LEFT JOIN (all users, even without orders) SELECT u.name, COALESCE(o.total, 0) as total FROM users u LEFT JOIN orders o ON u.id = o.user_id; -- Multiple joins SELECT u.name, o.total, p.name as product FROM users u JOIN orders o ON u.id = o.user_id JOIN order_items oi ON o.id = oi.order_id JOIN products p ON oi.product_id = p.id; -- Self join (hierarchical data) SELECT e.name as employee, m.name as manager FROM employees e LEFT JOIN employees m ON e.manager_id = m.id;

Subqueries & CTEs

-- Subquery in WHERE SELECT * FROM users WHERE id IN ( SELECT user_id FROM orders WHERE total > 1000 ); -- Common Table Expression (CTE) WITH active_users AS ( SELECT * FROM users WHERE status = 'active' ), user_orders AS ( SELECT user_id, SUM(total) as total_spent FROM orders GROUP BY user_id ) SELECT au.name, COALESCE(uo.total_spent, 0) FROM active_users au LEFT JOIN user_orders uo ON au.id = uo.user_id; -- Recursive CTE (tree traversal) WITH RECURSIVE org_tree AS ( SELECT id, name, manager_id, 1 as level FROM employees WHERE manager_id IS NULL UNION ALL SELECT e.id, e.name, e.manager_id, t.level + 1 FROM employees e JOIN org_tree t ON e.manager_id = t.id ) SELECT * FROM org_tree;

Window Functions

-- Row number SELECT name, age, ROW_NUMBER() OVER (ORDER BY age DESC) as rank FROM users; -- Partition by SELECT name, department, salary, RANK() OVER (PARTITION BY department ORDER BY salary DESC) FROM employees; -- Running total SELECT date, amount, SUM(amount) OVER ( ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) as running_total FROM transactions; -- Lag/Lead SELECT date, amount, LAG(amount) OVER (ORDER BY date) as prev_amount, LEAD(amount) OVER (ORDER BY date) as next_amount FROM transactions;

🐘 PostgreSQL Indexing

Index Types

-- B-Tree (default, most common) CREATE INDEX idx_users_email ON users(email); -- Unique index CREATE UNIQUE INDEX idx_users_email_unique ON users(email); -- Composite index CREATE INDEX idx_orders_user_date ON orders(user_id, created_at DESC); -- Partial index CREATE INDEX idx_active_users ON users(email) WHERE status = 'active'; -- Expression index CREATE INDEX idx_lower_email ON users(LOWER(email)); -- GIN index (for JSONB, arrays, full-text) CREATE INDEX idx_users_metadata ON users USING GIN(metadata); -- GiST index (for geometric, full-text) CREATE INDEX idx_locations_point ON locations USING GIST(point); -- Hash index (equality only, rarely used) CREATE INDEX idx_users_id_hash ON users USING HASH(id);

Index Analysis

-- Explain query plan EXPLAIN SELECT * FROM users WHERE email = 'test@test.com'; -- Explain with actual execution EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'test@test.com'; -- Check index usage SELECT schemaname, tablename, indexname, idx_scan as scans, idx_tup_read as tuples_read, idx_tup_fetch as tuples_fetched FROM pg_stat_user_indexes ORDER BY idx_scan DESC; -- Find unused indexes SELECT schemaname, tablename, indexname FROM pg_stat_user_indexes WHERE idx_scan = 0; -- Drop index DROP INDEX idx_users_email;
💡 Indexing Tips:
  • Index columns in WHERE, JOIN, ORDER BY clauses
  • Order matters in composite indexes (leftmost prefix)
  • Don't over-index - slows down writes
  • Use EXPLAIN ANALYZE to verify index usage
  • Consider partial indexes for common filtered queries

🐘 PostgreSQL JSONB Operations

JSONB Insert & Query

-- Insert JSON data INSERT INTO users (email, metadata) VALUES ( 'john@example.com', '{"preferences": {"theme": "dark", "lang": "en"}, "tags": ["premium", "beta"]}'::jsonb ); -- Extract value SELECT metadata->'preferences'->'theme' FROM users; SELECT metadata->>'preferences' FROM users; -- as text -- Query JSON data SELECT * FROM users WHERE metadata->'preferences'->>'theme' = 'dark'; -- Contains operator SELECT * FROM users WHERE metadata @> '{"preferences": {"theme": "dark"}}'; -- Check if key exists SELECT * FROM users WHERE metadata ? 'preferences'; -- Array contains SELECT * FROM users WHERE metadata->'tags' ? 'premium';

JSONB Modification

-- Update JSON field UPDATE users SET metadata = metadata || '{"preferences": {"theme": "light"}}'::jsonb WHERE id = 1; -- Delete key UPDATE users SET metadata = metadata - 'old_key'; -- Set nested value UPDATE users SET metadata = jsonb_set( metadata, '{preferences,theme}', '"light"'::jsonb ); -- Add to array UPDATE users SET metadata = jsonb_set( metadata, '{tags}', metadata->'tags' || '"new_tag"'::jsonb );

🌍 Cassandra Tables & Data Modeling

Table Creation

-- Create table with primary key CREATE TABLE users ( user_id UUID PRIMARY KEY, email TEXT, name TEXT, age INT, created_at TIMESTAMP ); -- Composite primary key (partition + clustering) CREATE TABLE orders ( user_id UUID, -- Partition key order_date TIMESTAMP, -- Clustering key order_id UUID, total DECIMAL, status TEXT, PRIMARY KEY ((user_id), order_date) ) WITH CLUSTERING ORDER BY (order_date DESC); -- Multiple clustering columns CREATE TABLE events ( device_id UUID, -- Partition key event_date DATE, -- Clustering key 1 event_time TIMESTAMP, -- Clustering key 2 event_type TEXT, data TEXT, PRIMARY KEY ((device_id), event_date, event_time) ) WITH CLUSTERING ORDER BY (event_date ASC, event_time DESC);
🔑 Primary Key Concepts:
  • Partition Key: Determines which node stores the data (hash distribution)
  • Clustering Key: Determines sort order within a partition
  • Composite Partition Key: Multiple columns in partition key: ((col1, col2), col3)

Data Modeling Principles

📐 Query-First Design:
  • Design tables based on queries, not entities
  • One table per query pattern (denormalization is OK)
  • Partition for even data distribution
  • Cluster for query ordering
  • Avoid ALLOW FILTERING

Example: Time Series Data

-- Sensor readings (time series pattern) CREATE TABLE sensor_data ( sensor_id UUID, -- Partition: one sensor's data together reading_date DATE, -- Clustering: group by day reading_time TIMESTAMP, -- Clustering: time ordered temperature DOUBLE, humidity DOUBLE, PRIMARY KEY ((sensor_id), reading_date, reading_time) ) WITH CLUSTERING ORDER BY (reading_date ASC, reading_time ASC); -- Efficient query: Get sensor data for a date range SELECT * FROM sensor_data WHERE sensor_id = ? AND reading_date >= ? AND reading_date <= ?;

🌍 Cassandra CRUD Operations

Insert

-- Basic insert INSERT INTO users (user_id, email, name, age) VALUES (uuid(), 'john@example.com', 'John Doe', 30); -- Insert with TTL (time-to-live) INSERT INTO sessions (session_id, user_id, data) VALUES (?, ?, ?) USING TTL 3600; -- Expires in 1 hour -- Insert with timestamp (for conflict resolution) INSERT INTO users (user_id, name) VALUES (?, ?) USING TIMESTAMP 1234567890; -- Insert JSON INSERT INTO users JSON '{ "user_id": "uuid-here", "email": "test@example.com", "name": "Test User" }';

Select

-- Must include partition key for efficient query SELECT * FROM orders WHERE user_id = ?; -- With clustering key range SELECT * FROM orders WHERE user_id = ? AND order_date >= '2024-01-01' AND order_date < '2024-02-01'; -- Limit results SELECT * FROM orders WHERE user_id = ? LIMIT 10; -- Allow filtering (AVOID in production!) SELECT * FROM users WHERE email = 'test@example.com' ALLOW FILTERING; -- Scans all partitions

Update

-- Update (upsert behavior) UPDATE users SET name = 'John Smith', age = 31 WHERE user_id = ?; -- Update with conditions UPDATE users SET name = 'New Name' WHERE user_id = ? IF name = 'Old Name'; -- Lightweight transaction -- Update collection UPDATE users SET tags = tags + {'premium'} WHERE user_id = ?; -- Increment counter UPDATE page_views SET view_count = view_count + 1 WHERE page_id = ?;

Delete

-- Delete row DELETE FROM users WHERE user_id = ?; -- Delete specific columns DELETE age, name FROM users WHERE user_id = ?; -- Delete from collection UPDATE users SET tags = tags - {'premium'} WHERE user_id = ?; -- Truncate table TRUNCATE users;

🌍 Cassandra Collections & Advanced Types

Collection Types

-- Table with collections CREATE TABLE products ( product_id UUID PRIMARY KEY, name TEXT, tags SET<TEXT>, -- Unique values features LIST<TEXT>, -- Ordered, duplicates allowed metadata MAP<TEXT, TEXT> -- Key-value pairs ); -- Insert with collections INSERT INTO products (product_id, name, tags, features, metadata) VALUES ( uuid(), 'Laptop', {'electronics', 'computers'}, ['16GB RAM', '512GB SSD'], {'brand': 'Dell', 'model': 'XPS'} ); -- Update SET UPDATE products SET tags = tags + {'gaming'} WHERE product_id = ?; UPDATE products SET tags = tags - {'computers'} WHERE product_id = ?; -- Update LIST UPDATE products SET features = features + ['Backlit KB'] WHERE product_id = ?; UPDATE products SET features = ['New Feature'] + features WHERE product_id = ?; UPDATE products SET features[0] = '32GB RAM' WHERE product_id = ?; -- Update MAP UPDATE products SET metadata['warranty'] = '2 years' WHERE product_id = ?; UPDATE products SET metadata = metadata + {'color': 'silver'} WHERE product_id = ?; DELETE metadata['model'] FROM products WHERE product_id = ?;

User-Defined Types (UDT)

-- Create type CREATE TYPE address ( street TEXT, city TEXT, state TEXT, zip TEXT ); -- Use UDT in table CREATE TABLE customers ( customer_id UUID PRIMARY KEY, name TEXT, addresses LIST<FROZEN<address>> ); -- Insert with UDT INSERT INTO customers (customer_id, name, addresses) VALUES (uuid(), 'John', [ {street: '123 Main', city: 'NYC', state: 'NY', zip: '10001'} ]);

Counter Tables

-- Counter table (only counters allowed) CREATE TABLE page_views ( page_id UUID PRIMARY KEY, view_count COUNTER ); -- Increment counter UPDATE page_views SET view_count = view_count + 1 WHERE page_id = ?; -- Decrement counter UPDATE page_views SET view_count = view_count - 1 WHERE page_id = ?;

🎯 When to Use What?

🐘 Choose PostgreSQL When:

  • Complex queries with JOINs required
  • ACID compliance is critical (financial, inventory)
  • Data relationships are important
  • Need for ad-hoc reporting and analytics
  • Moderate data volume (GB to TB)
  • Strong consistency is required
  • Full-text search on structured data
  • GIS/spatial data (PostGIS extension)
  • JSON with complex queries (JSONB)
  • Complex business logic in database

Common Use Cases:

  • Financial applications
  • E-commerce platforms
  • CRM and ERP systems
  • Content management systems
  • Analytics and reporting

🌍 Choose Cassandra When:

  • Massive write throughput needed
  • Petabyte-scale data storage
  • Linear horizontal scaling required
  • Multi-datacenter replication
  • High availability with no downtime
  • Time-series data (IoT, logs, metrics)
  • Simple queries known in advance
  • Eventual consistency is acceptable
  • Geo-distributed deployments
  • Write-heavy workloads

Common Use Cases:

  • IoT sensor data
  • Time-series metrics
  • Message/Chat applications
  • Recommendation engines
  • Audit logs and event tracking

Query Optimization

-- Analyze table statistics ANALYZE users; -- Vacuum (reclaim space) VACUUM users; VACUUM FULL users; -- Full table rewrite VACUUM ANALYZE users; -- Reindex REINDEX TABLE users; REINDEX INDEX idx_users_email; -- Find slow queries SELECT query, calls, total_time, mean_time FROM pg_stat_statements ORDER BY total_time DESC LIMIT 10; -- Find table bloat SELECT schemaname, tablename, pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) as size FROM pg_tables ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC;

🌍 Cassandra Replication & Consistency

Consistency Levels

LevelWriteReadDescription
ANY✓-Write to any node (hinted handoff)
ONE✓✓Nearest replica responds
TWO✓✓Wait for 2 replicas
THREE✓✓Wait for 3 replicas
QUORUM✓✓Majority (RF/2 + 1)
ALL✓✓All replicas must respond
LOCAL_QUORUM✓✓Quorum in local DC
EACH_QUORUM✓-Quorum in each DC

Setting Consistency

-- Set consistency for session CONSISTENCY QUORUM; -- Per query consistency SELECT * FROM users WHERE user_id = ? USING CONSISTENCY LOCAL_QUORUM; -- Write with consistency INSERT INTO users (user_id, name) VALUES (?, ?) USING CONSISTENCY QUORUM AND TTL 86400;

Replication Factor Trade-offs

📊 Replication Factor (RF) Guidelines:
  • RF=1: No fault tolerance (single node failure = data loss)
  • RF=2: Can survive 1 node failure, but no reads during failure with QUORUM
  • RF=3: Recommended minimum, can survive 1 failure with QUORUM
  • RF=5: Can survive 2 failures with QUORUM, higher availability

Consistency vs Availability

# Strong Consistency Formula R + W > RF # Where: R = read consistency, W = write consistency, RF = replication factor # Example: RF=3, QUORUM read (R=2), QUORUM write (W=2) 2 + 2 = 4 > 3 ✓ (Strong consistency guaranteed) # Eventual Consistency R + W <= RF # Example: RF=3, ONE read (R=1), ONE write (W=1) 1 + 1 = 2 <= 3 (Eventual consistency, higher availability)

✅ Best Practices Comparison

🐘 PostgreSQL Best Practices

  • Use appropriate data types (don't over-size)
  • Create indexes on frequently queried columns
  • Use EXPLAIN ANALYZE for query optimization
  • Implement connection pooling (PgBouncer)
  • Regular VACUUM and ANALYZE
  • Use transactions for related operations
  • Implement proper constraints (NOT NULL, CHECK)
  • Use JSONB for flexible schemas
  • Set up replication for HA
  • Monitor with pg_stat_statements
  • Use prepared statements to prevent SQL injection
  • Partition large tables

🌍 Cassandra Best Practices

  • Design tables based on queries (query-first)
  • Use appropriate partition key for even distribution
  • Avoid hot partitions (time-based keys)
  • Keep partitions under 100MB
  • Denormalize data (one table per query)
  • Use appropriate consistency level per operation
  • Set TTL for time-based data expiration
  • Avoid ALLOW FILTERING
  • Run regular repairs (nodetool repair)
  • Monitor tombstones and compaction
  • Use prepared statements for performance
  • Choose right compaction strategy

❌ Common Anti-Patterns to Avoid

🐘 PostgreSQL Anti-Patterns:
  • ❌ SELECT * in production queries
  • ❌ Missing WHERE clause on DELETE/UPDATE
  • ❌ Not using connection pooling
  • ❌ Over-indexing (slows writes)
  • ❌ Not using EXPLAIN before queries
  • ❌ Ignoring VACUUM maintenance
  • ❌ Using TEXT for everything
  • ❌ Not setting up proper backups
  • ❌ Storing large files in database (use S3)
🌍 Cassandra Anti-Patterns:
  • ❌ Using ALLOW FILTERING in production
  • ❌ Hot partitions (all data in one partition)
  • ❌ Very large partitions (>100MB)
  • ❌ Secondary indexes on high-cardinality columns
  • ❌ Thinking like relational (joins, normalization)
  • ❌ Not running regular repairs
  • ❌ Too many columns in primary key
  • ❌ Ignoring tombstone accumulation
