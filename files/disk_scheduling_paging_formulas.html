💾 Disk Scheduling Algorithms

Basic Disk Scheduling Formulas

Total Head Movement
Total Head Movement = Σ |Current Position - Next Position|
The sum of absolute differences between consecutive disk positions that the read/write head traverses.
Variables: Current Position = Current cylinder position, Next Position = Next cylinder to be accessed
Average Seek Time
Average Seek Time = Total Head Movement / Number of Requests
The average time taken for the disk arm to move to the specified track for all requests.
📌 Example:
If Total Head Movement = 500 cylinders and Number of Requests = 10
Average Seek Time = 500 / 10 = 50 cylinders
Total Seek Time
Total Seek Time = Total Head Movement × Time per Cylinder
Converts the head movement distance into actual time units based on the disk's seek time characteristics.
Disk Service Time
Service Time = Seek Time + Rotational Latency + Transfer Time
Where:
• Seek Time = Time to move arm to correct cylinder
• Rotational Latency = Time for sector to rotate under head (~ half rotation)
• Transfer Time = Time to read/write data

FCFS (First Come First Serve)

Algorithm Steps

Requests are serviced in the order they arrive in the queue
No reordering or optimization of requests
Simple but may cause long seek times
Calculate total head movement by summing distances between consecutive requests

Formula

Head Movement = |Head - R₁| + |R₁ - R₂| + |R₂ - R₃| + ... + |Rₙ₋₁ - Rₙ|
Where: R₁, R₂, ..., Rₙ are requests in arrival order, Head is initial position
⚠️ Note:
FCFS is fair but not optimal. Can cause excessive head movement (thrashing) if requests are scattered.

SSTF (Shortest Seek Time First)

Algorithm Steps

Select the request with minimum seek time from current position
Calculate distance to all pending requests
Service the closest request first
Repeat until all requests are serviced

Selection Formula

Next Request = argmin(|Current Position - Rᵢ|)
Select request Rᵢ that minimizes the absolute difference from current head position.
⚠️ Note:
May cause starvation for distant requests. Better throughput than FCFS but less fair.

SCAN (Elevator Algorithm)

Algorithm Steps

Head moves in one direction servicing all requests
When end is reached, direction reverses
Continue servicing requests in reverse direction
Like an elevator - goes to top, then comes down

Formula (Moving Towards End)

If Head → End: Movement = (End - Head) + (End - Min_Request)
If Head → Start: Movement = (Head - Start) + (Max_Request - Start)
Where:
• End = Last cylinder (e.g., 199 or 4999)
• Start = First cylinder (usually 0)
• Min_Request = Smallest pending request
• Max_Request = Largest pending request

C-SCAN (Circular SCAN)

Algorithm Steps

Head moves in one direction only (e.g., towards end)
Services all requests on the way
When end is reached, jump to start (circular)
Continue in same direction from start

Formula

Movement = (End - Head) + End + Max_Request
Or = Total Cylinders - 1 + (End - Head) + Max_Request
Accounts for the jump from end to start as part of total movement.
💡 Key Point:
More uniform wait time than SCAN as requests at beginning don't have to wait for return trip.

LOOK & C-LOOK

LOOK Algorithm

Movement = (Max_Request - Head) + (Max_Request - Min_Request)
Similar to SCAN but doesn't go to the end. Reverses direction at the last request in current direction.
Go only as far as the last request
Reverse direction at extreme request
No need to reach disk boundaries

C-LOOK Algorithm

Movement = (Max_Request - Head) + (Max_Request - Min_Request) + (Min_Request - Head)
Combines C-SCAN and LOOK. Moves in one direction, jumps from last request to first, continues.
Like C-SCAN but stops at extreme requests
Jumps from max to min request
More efficient than C-SCAN

Algorithm Comparison

Algorithm Throughput Avg. Seek Time Starvation Variance
FCFS Low High No High
SSTF High Low Yes High
SCAN Medium Medium No Medium
C-SCAN Medium Medium No Low
LOOK Medium-High Medium-Low No Medium
C-LOOK High Low No Low

📄 Paging Formulas

Basic Paging Calculations

Number of Pages
Number of Pages = Logical Address Space / Page Size
Number of Pages = 2^(Logical Address Bits - Page Offset Bits)
📌 Example:
Logical Address Space = 64 KB, Page Size = 4 KB
Number of Pages = 64 KB / 4 KB = 16 pages
Number of Frames
Number of Frames = Physical Address Space / Frame Size
Number of Frames = 2^(Physical Address Bits - Frame Offset Bits)
Note: Frame Size = Page Size (pages and frames are same size)
Page Table Size
Page Table Size = Number of Pages × Entry Size
If multi-level: Level-1 Size + Level-2 Size + ...
📌 Example:
16 pages, Entry Size = 4 bytes
Page Table Size = 16 × 4 = 64 bytes

Address Translation

Logical Address Breakdown
Page Number (p) = ⌊Logical Address / Page Size⌋
Page Offset (d) = Logical Address mod Page Size
Or in binary: Page Number | Page Offset
Where:
• Page Number bits = log₂(Number of Pages)
• Page Offset bits = log₂(Page Size)
Physical Address Calculation
Physical Address = Frame Number × Page Size + Offset
Or: Physical Address = (Frame Number << Offset Bits) | Offset
📌 Example:
Frame Number = 5, Page Size = 4 KB, Offset = 100
Physical Address = 5 × 4096 + 100 = 20580
Address Translation Formula
PA = [PageTable[p] × PageSize] + d
Where:
• PA = Physical Address
• p = Page Number from logical address
• PageTable[p] = Frame Number at page table entry p
• d = Page Offset

Bit Calculations

Address Structure

Logical Address = Page Number Bits + Offset Bits
Physical Address = Frame Number Bits + Offset Bits
Key Point: Offset bits are the same for both logical and physical addresses because page size equals frame size.

Number of Bits

Page Number Bits = ⌈log₂(Number of Pages)⌉
Frame Number Bits = ⌈log₂(Number of Frames)⌉
Offset Bits = log₂(Page Size)

Page Fault & Memory Access

Effective Access Time (EAT)
EAT = (1 - p) × Memory Access Time + p × Page Fault Time
EAT = (1 - p) × ma + p × (Page Fault Overhead + Swap Page + Restart)
Where:
• p = Page Fault Rate (0 ≤ p ≤ 1)
• ma = Memory Access Time (nanoseconds)
• Page Fault Time = Time to handle page fault (milliseconds)
📌 Example:
p = 0.001, ma = 100 ns, Page Fault Time = 8 ms
EAT = (1 - 0.001) × 100 + 0.001 × 8,000,000
EAT = 99.9 + 8000 = 8099.9 ns
Page Fault Time Components
Page Fault Time = Service Page Fault + Read Page + Restart Process
Components:
• Service Page Fault: OS overhead to handle the trap
• Read Page: Disk I/O to read the page (major component)
• Restart Process: Context switch back to process
Page Fault Rate Calculation
Maximum p such that EAT ≤ Acceptable Time
p_max = (EAT_max - ma) / (Page Fault Time - ma)
📌 Example:
If acceptable EAT = 200 ns, ma = 100 ns, Page Fault Time = 10 ms
p_max = (200 - 100) / (10,000,000 - 100) ≈ 0.00001

TLB (Translation Lookaside Buffer)

EAT with TLB
EAT = TLB Hit Ratio × (TLB Access + Memory Access) + (1 - TLB Hit Ratio) × (TLB Access + Page Table Access + Memory Access)
Simplified: EAT = TLB Access + Hit Ratio × ma + (1 - Hit Ratio) × 2ma
Assumptions:
• TLB Access time is typically included in memory access
• Page table lookup requires one memory access
• Final data access requires one memory access
Complete EAT Formula with TLB and Page Fault
EAT = TLB_access + (1-p)[hit×ma + (1-hit)×2ma] + p×PageFaultTime
Where:
• TLB_access = Time to search TLB
• hit = TLB hit ratio
• p = Page fault rate
• ma = Memory access time

Page Replacement Algorithms

FIFO (First In First Out)

Replace the page that has been in memory longest
Maintain a queue of pages
On page fault, remove first page in queue
Add new page at end of queue
⚠️ Belady's Anomaly:
More frames can cause MORE page faults!

Optimal (OPT/MIN)

Replace page that will be used farthest in future
Look ahead in reference string
Find page not needed for longest time
Theoretical best - not implementable

LRU (Least Recently Used)

Replace page that has not been used for longest time
Track when each page was last accessed
On fault, remove least recently used page
Approximation of optimal algorithm

Page Fault Rate

Page Fault Rate = Number of Page Faults / Total References
📌 Example:
Reference string: 7, 0, 1, 2, 0, 3, 0, 4, 2, 3
3 frames: Count page faults
Lower rate = Better performance

Thrashing

CPU Utilization & Degree of Multiprogramming
If CPU utilization < Threshold, reduce degree of multiprogramming
Thrashing occurs when a process spends more time paging than executing. The system degrades rapidly as processes compete for frames.
Working Set Model
Working Set = {pages accessed in last Δ references}
Total Frames Needed ≥ Σ Working Set Size for all processes
Where: Δ = Working set window parameter
If total frames needed > available frames → Thrashing occurs
Page Fault Frequency (PFF)
If PFF > Upper Threshold → Allocate more frames
If PFF < Lower Threshold → Reduce frames allocated
PFF is a local replacement algorithm that adjusts frame allocation based on page fault frequency.

Memory Management Calculations

Internal Fragmentation
Internal Fragmentation = Page Size - (Actual Data Size mod Page Size)
Average Internal Fragmentation = Page Size / 2
Wasted space within the last partially filled page of a process.
External Fragmentation (Not in Paging)
External Fragmentation = 0 (in pure paging)
Paging eliminates external fragmentation because pages are fixed size. However, internal fragmentation still exists.
Multi-level Page Table
Outer Page Table Size = 2^(Outer Index Bits) × Entry Size
Inner Page Tables = Number of valid outer entries
Total Space = Outer Page Table + Active Inner Tables
📌 Example:
32-bit address, 4KB pages, 4 bytes per entry
• Offset = 12 bits
• 2-level: Outer = 10 bits, Inner = 10 bits
• Each page table fits in one page

Page#: Address / PageSize
