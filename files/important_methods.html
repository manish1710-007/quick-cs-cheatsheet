🔤 STRING METHODS
#01 isEmpty() vs. isBlank() BUG TRAP
isEmpty()
True only if length == 0. Treats spaces as valid.
isBlank()
True if empty OR whitespace-only (spaces, tabs, newlines).
// WHEN TO USE Use isEmpty() when whitespace is meaningful (passwords, file formats). Use isBlank() for almost all user input validation — users accidentally press spacebar constantly.
JAVA
String input = " ";
input.isEmpty();   // false — has length
input.isBlank();   // true  — safe to reject
#02 + (Concat) vs. StringBuilder.append() PERFORMANCE
+ operator
Creates a NEW String object on every use. Massive GC pressure in loops.
append()
Mutates an in-memory buffer. O(1) amortized. One object, always.
// WHEN TO USE + for single-line log statements (compiler auto-optimizes). StringBuilder inside any loop or dynamic string construction.
JAVA
// BAD: 1000 String objects
String s = "";
for(int i=0; i<1000; i++) s += i;

// GOOD: 1 buffer
StringBuilder sb = new StringBuilder();
for(int i=0; i<1000; i++) sb.append(i);
⚖️ EQUALITY & IDENTITY
#03 == vs. .equals() BUG TRAP
==
Compares memory address. Are these the same object?
.equals()
Compares value/content. Do these hold the same data?
// WHEN TO USE == for Enums and primitives (int, char, boolean). .equals() for all Objects — String, Integer, custom classes. Always.
JAVA
String a = new String("text");
String b = new String("text");
a == b          // false — different heap addresses
a.equals(b)   // true  — same content
#04 equals() vs. hashCode() Contract BUG TRAP
equals()
Logical equality — same content?
hashCode()
Integer digest for bucketing in HashMap/HashSet.
// ⚠ GOLDEN RULE If you override equals(), you MUST override hashCode(). Equal objects must share the same hash. Violating this causes HashSet to store duplicates and HashMap to lose keys silently.
// WHEN TO USE Always override both together via IDE or Lombok @EqualsAndHashCode. Never override just one.
#05 getClass() vs. .class REFLECTION
.class
Compile-time. No object needed. Use when type is known statically.
getClass()
Runtime. Reveals true dynamic type through polymorphism.
JAVA
Class<String> c1 = String.class;  // compile time
Object obj = "Hello";
Class<?> c2 = obj.getClass(); // → String.class (runtime)
🧵 THREADING & CONCURRENCY
#06 sleep() vs. wait() CONCURRENCY
sleep()
Thread naps but KEEPS the lock. Does not release resources.
wait()
Thread releases the lock and waits for notify()/notifyAll().
// ⚠ DEADLOCK RISK Using sleep() inside a synchronized block holds the lock — other threads can never enter. Classic deadlock.
JAVA
synchronized(queue) {
  while(queue.isEmpty()) {
    queue.wait();   // releases lock — producer can add
    // Thread.sleep(1000) — WRONG: holds lock → deadlock
  }
}
#07 Thread.start() vs. Thread.run() CONCURRENCY
run()
Plain method call. Runs in the CURRENT thread. No parallelism.
start()
Spawns a new OS thread, then calls run() inside it.
// ⚠ COMMON MISTAKE Calling t.run() makes "multithreaded" code run sequentially. UI freezes, tasks block. Always use start().
JAVA
Thread t = new Thread(() -> doWork());
t.run();    // WRONG — runs in main thread
t.start(); // CORRECT — new thread spawned
#08 notify() vs. notifyAll() CONCURRENCY
notify()
Wakes ONE random waiting thread. May wake the wrong type.
notifyAll()
Wakes ALL waiting threads. Let them compete for the lock safely.
// WHEN TO USE Default to notifyAll() in 99% of cases. Use notify() only if you can guarantee exactly one thread is waiting and it can handle any notification.

#09 List.remove(index) vs. remove(Object) BUG TRAP
// ⚠ DANGEROUS AMBIGUITY On a List<Integer>, remove(5) removes the element at index 5 — NOT the value 5. If the list has <6 elements, you get IndexOutOfBoundsException.
JAVA
List<Integer> nums = new ArrayList<>(List.of(5, 10));
nums.remove(5);                      // IndexOutOfBoundsException!

#10 Comparable vs. Comparator SORTING
Comparable
Inside the class. One natural sort order. compareTo().
Comparator
Outside the class. Unlimited sort strategies. Passed at call site.
JAVA
class User implements Comparable<User> {
  public int compareTo(User u) { return this.id - u.id; }
}
// Custom — outside the class
users.sort(Comparator.comparing(User::getName));

#11 flush() vs. close() DATA LOSS
flush()
Pushes buffer to disk/network. Stream stays OPEN for more writes.
close()
Flushes + releases OS resources. Stream is DONE, cannot reuse.
// WHEN TO USE flush() during long-running writes (logs, ZIPs) for periodic safety. close() when finished. Use try-with-resources — it auto-closes and prevents resource leaks.
#12 exists() vs. isFile() BUG TRAP
exists()
True if path exists — could be a file, directory, or symlink.
isFile()
True only if it exists AND is a regular file (not a directory).
// ⚠ THE TRAP A directory named data.txt makes exists() return true. Attempting to read it as a file will crash. Always use isRegularFile() before reading.
JAVA
// if(Files.exists(path)) Files.readAllLines(path); — DANGEROUS
if(Files.isRegularFile(path)) {
  Files.readAllLines(path);   // SAFE
}

#15 floor() vs. ceil() vs. round() MATH
floor(x)
Always rounds DOWN. 2.9 → 2, -2.1 → -3.
ceil(x)
Always rounds UP. 2.1 → 3, -2.9 → -2.
// WHEN TO USE floor() for grid/tile alignment. ceil() for pagination: 101 items ÷ 10 per page = ceil(10.1) = 11 pages. round() for display values.

array.length
Property (field). No parentheses. Arrays only.
str.length()
Method call. Strings only. Has parentheses.
JAVA
int[] arr = {1,2,3};     arr.length    // array → field
String s = "hi";         s.length()   // string → method
List<String> l = ...;   l.size()     // collection → method
⚡ BONUS METHODS
#17 Optional.orElse() vs. orElseGet() PERFORMANCE
orElse(T)
The fallback expression is ALWAYS evaluated — even if value is present.
orElseGet(Supplier)
Lazy — fallback only evaluated if value is absent.
// ⚠ SILENT PERF KILLER orElse(expensiveCall()) always runs expensiveCall() — even when the Optional has a value. Use orElseGet(this::expensiveCall) to defer it.
JAVA
// Always calls createDefault() — even if user exists!
user.orElse(createDefault());
// Lazy: only calls if user is absent
user.orElseGet(this::createDefault);
#18 Iterator.remove() vs. List.remove() in Loops BUG TRAP
// ⚠ ConcurrentModificationException Calling list.remove() inside a for-each loop throws ConcurrentModificationException. The collection detects structural modification while iterating.
JAVA
// for(String s : list) list.remove(s); — THROWS CME

// Safe: use Iterator
Iterator<String> it = list.iterator();
while(it.hasNext()) {
  if(condition(it.next())) it.remove(); // safe
}
// Or Java 8+: list.removeIf(condition);
#19 String.format() vs. .formatted() (Java 15+) MODERN JAVA
String.format()
Static method. Works everywhere (Java 5+). Verbose to call.
.formatted()
Instance method on String literals. Cleaner syntax. Java 15+.
JAVA
// Old style
String s1 = String.format("Hello %s, you are %d", name, age);
// Modern — inline on the template itself
String s2 = "Hello %s, you are %d".formatted(name, age);
#20 unmodifiableList() vs. List.copyOf() DEFENSIVE COPY
unmodifiableList()
A READ-ONLY VIEW. Mutations to the original list are still reflected!
List.copyOf()
A true immutable snapshot. Unaffected by future changes to source.
// ⚠ SNEAKY MUTATION BUG If you return Collections.unmodifiableList(internalList), callers can't modify it — but YOU can still mutate the internal list and they'll see the changes. Use List.copyOf() for a true defensive copy.
JAVA
List<String> src = new ArrayList<>(List.of("a","b"));
List<String> view = Collections.unmodifiableList(src);
src.add("c");
view.size(); // 3 — still sees the mutation!

List<String> safe = List.copyOf(src);  // true snapshot
🚨 EXCEPTION HANDLING
#21 Checked vs. Unchecked Exceptions EXCEPTIONS
Checked (Exception)
Compiler forces you to handle or declare. Recoverable scenarios (IO, DB).
Unchecked (RuntimeException)
Not enforced by compiler. Programming errors (NPE, ArrayIndexOutOfBounds).
// WHEN TO USE Throw checked exceptions when the caller can realistically recover (file not found → ask user to retry). Throw unchecked for bugs that should never happen in correct code (null where impossible).
JAVA
// Checked — caller MUST handle
public void readFile(String path) throws IOException { ... }

// Unchecked — programming error
if(id < 0) throw new IllegalArgumentException("ID negative");
#22 finally vs. try-with-resources RESOURCE LEAK
finally
Always runs — but you must manually close resources. Easy to forget.
try-with-resources
Auto-calls close() on any AutoCloseable. Cleaner, safer, preferred.
// ⚠ EXCEPTION SUPPRESSION IN FINALLY If both the try block AND the finally block throw, the finally exception silently swallows the original. try-with-resources handles this correctly via getSuppressed().
JAVA
// Old — manual, risky
Connection c = getConn();
try { ... } finally { c.close(); }

// Modern — auto-close, exception-safe
try(Connection c = getConn()) {
  // c.close() called automatically
}
#23 catch(Exception) vs. catch(Throwable) BUG TRAP
catch(Exception)
Catches all checked + unchecked exceptions. Misses Errors.
catch(Throwable)
Catches everything — including OutOfMemoryError, StackOverflowError.
// ⚠ NEVER CATCH ERRORS Error subclasses (OOM, SOE) mean the JVM is in an unrecoverable state. Catching them and continuing leads to silent corruption. Let them propagate — or log and exit.
JAVA
// DANGEROUS — hides OOM, SOE
catch(Throwable t) { log.error("oops"); }

// Correct — only catch what you can handle
catch(IOException e) { retry(); }
catch(SQLException e) { rollback(); }
#24 throw vs. throws SYNTAX
throw
Statement that actually fires an exception at runtime.
throws
Declaration on a method signature — warns callers it might propagate.
JAVA
// throws = declaration (callers beware)
public void parse(String s) throws ParseException {
  if(s == null)
    throw new ParseException("null input", 0); // throw = fires it
}
#25 Multi-catch vs. Stacked catch Blocks CLEAN CODE
Stacked catch
Separate block per exception. Verbose when logic is identical.
Multi-catch (Java 7+)
Single block for multiple types via |. Concise, no duplication.
// ⚠ MULTI-CATCH MAKES e EFFECTIVELY FINAL Inside a multi-catch block you cannot reassign e. The compiler enforces this because the type is ambiguous.
JAVA
// Verbose — same body duplicated
catch(IOException e)     { log(e); }
catch(SQLException e)    { log(e); }

// Clean — Java 7+ multi-catch
catch(IOException | SQLException e) { log(e); }
🗺️ COLLECTIONS & MAPS
#26 HashMap vs. LinkedHashMap vs. TreeMap MAP CHOICE
HashMap
O(1) ops. No guaranteed order. Default choice.
LinkedHashMap
O(1) ops. Insertion order preserved. Good for LRU caches.
TreeMap
O(log n) ops. Sorted by key (natural or Comparator). Use for range queries.
JAVA
// Need sorted keys? TreeMap
TreeMap<String,Integer> scores = new TreeMap<>();
scores.firstKey();  scores.subMap("A", "M");

// Need insertion order? LinkedHashMap
Map<String,Integer> lru = new LinkedHashMap<>(16,0.75f,true);
#27 getOrDefault() vs. computeIfAbsent() MAP OPS
getOrDefault()
Returns a default if key absent. Does NOT store it in the map.
computeIfAbsent()
Computes and STORES the value if key absent. Lazy init.
// ⚠ COMMON GROUPING BUG Using getOrDefault(key, new ArrayList()) then modifying it does NOT update the map. You're modifying a detached list. Use computeIfAbsent instead.
JAVA
// BUG — list not stored in map
map.getOrDefault(key, new ArrayList<>()).add(val);

// CORRECT — creates & stores on first access
map.computeIfAbsent(key, k -> new ArrayList<>()).add(val);
#28 putIfAbsent() vs. merge() MAP OPS
putIfAbsent()
Inserts only if key not present. Returns old value (or null).
merge()
Inserts if absent; applies a function to combine old + new value if present.
// WHEN TO USE merge() is the go-to for frequency counters and aggregation. It eliminates the classic if(map.containsKey(k)) boilerplate.
JAVA
// Word frequency counter — old way
map.put(word, map.getOrDefault(word,0) + 1);

// Clean with merge
map.merge(word, 1, Integer::sum);
#29 ArrayDeque vs. LinkedList as Queue/Stack PERFORMANCE
LinkedList
Node-per-element. High GC overhead. Implements Queue + Deque.
ArrayDeque
Array-backed ring buffer. Faster, less GC, no null support. Preferred.
// WHEN TO USE Always prefer ArrayDeque when using a queue or stack. Java's own docs recommend it over Stack and over LinkedList for queue use. Only use LinkedList if you need null elements.
JAVA
// Avoid for stack/queue use
Stack<Integer> s = new Stack<>();
Queue<Integer> q = new LinkedList<>();

// Prefer — faster, less overhead
Deque<Integer> dq = new ArrayDeque<>();
#30 EnumSet vs. HashSet for Enum Values PERFORMANCE
HashSet<MyEnum>
General purpose. Works, but wasteful for enums. Uses hashing.
EnumSet
Backed by a single long bitmask. Extremely fast. Zero GC overhead.
// WHEN TO USE Any time you have a Set of enum values, use EnumSet. It's O(1) for all operations and uses a single primitive under the hood. No brainer.
JAVA
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

EnumSet<Day> weekend = EnumSet.of(Day.SAT, Day.SUN);
EnumSet<Day> workdays = EnumSet.complementOf(weekend);
🏗️ DESIGN PATTERNS & OOP
#31 Abstract Class vs. Interface OOP
Abstract Class
Can have state, constructors, concrete methods. Single inheritance only.
Interface
Contract-only (+ default methods). No state. A class can implement many.
// WHEN TO USE Abstract class for a shared base with common state/behavior (e.g., Animal with a name field). Interface for a capability contract that unrelated classes can implement (e.g., Serializable, Flyable). Prefer interfaces — they allow multiple inheritance of type.
#32 Composition vs. Inheritance DESIGN
Inheritance (is-a)
Tight coupling to parent. Fragile base class problem. Hard to change.
Composition (has-a)
Delegates to an owned object. Loosely coupled. Easier to swap behavior.
// ⚠ FRAGILE BASE CLASS Overriding a parent method that calls other methods in the parent can break silently when the parent changes. Composition avoids this entirely.
JAVA
// Inheritance — tight coupling
class LoggingList extends ArrayList<String> { ... }

// Composition — loosely coupled
class LoggingList {
  private final List<String> inner = new ArrayList<>();
  public void add(String s) { log(s); inner.add(s); }
}
#33 Singleton: Eager vs. Lazy Initialization CONCURRENCY
Eager
Instance created at class load. Thread-safe by JVM guarantee. Simple.
Lazy (Double-Checked)
Created on first use. Saves memory if rarely needed. Needs volatile.
// ⚠ MISSING volatile = BROKEN SINGLETON Without volatile, the JVM can reorder instructions: another thread may see a partially constructed object and use it. Always pair double-checked locking with volatile.
JAVA
// Eager — simplest, almost always fine
private static final Singleton INSTANCE = new Singleton();

// Lazy — must use volatile!
private static volatile Singleton instance;
if(instance == null) {
  synchronized(Singleton.class) {
    if(instance == null) instance = new Singleton();
  }
}
#34 Strategy vs. Template Method Pattern PATTERNS
Strategy
Inject an algorithm from outside at runtime. Composition-based.
Template Method
Define algorithm skeleton in base class; subclasses fill in steps. Inheritance-based.
// WHEN TO USE Strategy when you need to swap algorithms at runtime (sorting, payment methods). Template Method when the overall flow is fixed but a step varies (data parsers: open → parse → close, where parse differs).
JAVA
// Strategy — runtime swap
sorter.setStrategy(new QuickSort());
sorter.setStrategy(new MergeSort()); // swap anytime

// Template Method — skeleton fixed
abstract class Parser {
  final void run() { open(); parse(); close(); } // fixed
  abstract void parse();                              // varies
}
#35 Builder Pattern vs. Telescoping Constructors DESIGN
Telescoping Constructor
Multiple overloaded constructors. Unreadable for 4+ params. Error-prone.
Builder Pattern
Fluent API. Named setters. Optional fields clear. Immutable result.
JAVA
// Telescoping — what does 'true, false, 42' mean?
new User("Alice", "[email protected]", true, false, 42);

// Builder — self-documenting
new User.Builder("Alice")
  .email("[email protected]")
  .verified(true)
  .age(42)
  .build();
🧬 GENERICS & TYPE SYSTEM
#36 ? extends T vs. ? super T (PECS Rule) GENERICS
? extends T (Producer)
Read from it. Upper bounded. You can GET items out.
? super T (Consumer)
Write into it. Lower bounded. You can PUT items in.
// PECS MNEMONIC Producer Extends, Consumer Super. If you only read from a collection → extends. If you only write into it → super. If both → no wildcard.
JAVA
// READ from source — extends
void copy(List<? extends Number> src,
          List<? super Number> dst) {
  for(Number n : src) dst.add(n); // WRITE into dst — super
}
#37 Type Erasure & Why instanceof Fails GENERICS
Compile Time
Generic types fully visible. Type safety enforced.
Runtime (Bytecode)
Type parameters erased to raw types. List<String> becomes List.
// ⚠ CAN'T instanceof GENERIC TYPES At runtime, List<String> and List<Integer> are indistinguishable. Attempting obj instanceof List<String> is a compile error. Use instanceof List<?> or a typed check via reflection.
JAVA
// COMPILE ERROR — generic type erased at runtime
if(obj instanceof List<String>) { ... }

// Correct — wildcard works
if(obj instanceof List<?> list) {
  list.forEach(System.out::println);
}
#38 Raw Types vs. Bounded Wildcards GENERICS
Raw Type (List)
Bypasses all type safety. Heap pollution risk. Legacy only.
Wildcard (List<?>)
Unknown but safe. Compiler still enforces type constraints.
// ⚠ RAW TYPE HEAP POLLUTION Mixing raw types and generics can cause ClassCastException at runtime in completely unrelated code — far from the actual bug. Never use raw types in new code.
JAVA
List raw = new ArrayList();   // raw — unsafe, avoid
List<?> wild = new ArrayList<String>(); // wildcard — safe
List<String> typed = new ArrayList<>();  // specific — best
#39 Array Covariance vs. Generic Invariance TYPE SYSTEM
Arrays (Covariant)
String[] IS-A Object[]. Compiles fine. Can throw ArrayStoreException at runtime.
Generics (Invariant)
List<String> is NOT a List<Object>. Fails at compile time — safer.
JAVA
Object[] arr = new String[3]; // compiles!
arr[0] = 42;              // ArrayStoreException at runtime

List<Object> list = new ArrayList<String>(); // COMPILE ERROR — caught early
#40 Sealed Classes vs. Open Inheritance (Java 17+) MODERN JAVA
Open Class
Anyone can subclass. Flexible but uncontrolled hierarchies.
Sealed Class
Only explicitly permitted classes can extend. Enables exhaustive pattern matching.

sealed interface Shape
  permits Circle, Rect, Triangle {}

// Compiler enforces exhaustiveness in switch
double area = switch(shape) {
  case Circle c    -> Math.PI * c.r() * c.r();
  case Rect r      -> r.w() * r.h();
  case Triangle t  -> 0.5
