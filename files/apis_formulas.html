REST API Design

REST Principles

REST Constraints

  1. Client-Server: Separation of concerns
  2. Stateless: No session state on server
  3. Cacheable: Responses must be cacheable
  4. Uniform Interface: Standard methods & URIs
  5. Layered System: Hierarchical layers
  6. Code on Demand: Optional (JS)

Richardson Maturity Model

    ↑    HATEOAS (links in response)

    │    Individual URIs for resources
    │

Idempotency Explained

Idempotent: f(f(x)) = f(x)

Calling the same request multiple times produces the same result.

Examples:

GET /users/123: Always returns same user

PUT /users/123: Always sets user to same state

DELETE /users/123: After first delete, returns 404

POST /users: Each call creates new user (NOT idempotent)

Resource Naming Conventions

URL Naming Rules

  • Use nouns: /users, /orders, /products
  • Use plurals: /users (not /user)
  • Use lowercase: /user-profiles
  • Use hyphens: /order-items (not /orderItems)
  • No verbs in URL: Use HTTP methods
  • Hierarchical: /users/123/orders
  • Filtering: /users?status=active

Good vs Bad Examples

// ✅ GOOD - RESTful Design GET /users // List users GET /users/123 // Get user POST /users // Create user PUT /users/123 // Update user DELETE /users/123 // Delete user GET /users/123/orders // User's orders POST /users/123/orders // Create order // ❌ BAD - Non-RESTful GET /getUsers POST /createUser POST /deleteUser/123 GET /user-orders/123

JWT Security Best Practices

JWT Security Checklist

  • Algorithm: Use RS256 (asymmetric) or HS256 (symmetric)
  • Secret Key: 256+ bits for HS256
  • Expiration: Short-lived access tokens (15-60 min)
  • Validation: Verify signature, exp, iss, aud, iat
  • Storage: HttpOnly cookies or secure storage
  • HTTPS: Always use TLS

JWT Validation Steps

  1. Verify signature with secret/public key
  2. Check exp (not expired)
  3. Check iat (issued in the past)
  4. Check iss (issuer matches)
  5. Check aud (audience matches)
  6. Check nbf (not before)
  7. Check jti (not replayed)

Rate Limiting

Rate Limiting Algorithms

Token Bucket

Bucket (Capacity: b tokens)
Refill: r tokens/second

    ┌─────────────────────────────┐
    │ ● ● ● ● ● ○ ○ ○ ○ ○         │
    │     Current: 5              │
    │     Max: 10                 │
    └─────────────────────────────┘
           ▲
           │ Refill at rate r
           │
Request arrives:
  If tokens > 0: 
    - Consume 1 token
    - Allow request
  Else:
    - Reject request (429)

Parameters:
  b = bucket capacity (burst size)
  r = refill rate (sustained rate)

Leaky Bucket

Requests flow in at any rate
Processed at constant rate

       Requests
          │
          ▼
    ┌───────────┐
    │           │
    │  Bucket   │ ◄── Queue (FIFO)
    │           │
    └─────┬─────┘
          │
          ▼ Leaks at rate r
       Processed

Process rate = r requests/second
No bursts allowed (smooth traffic)

Fixed Window

Window: Fixed time intervals
Count requests per window

Time: │───Window 1───│───Window 2───│
Count:    [5]            [5]
          │              │
          ▼              ▼
      Limit: 10      Limit: 10

Problem: Burst at boundary
│───Window 1───│───Window 2───│
    [10 at end]    [10 at start]
    = 20 requests in short time!

Sliding Window Log

Track timestamps of each request
Count requests in sliding window

        Window slides with time
        ──────────────────────►
              │           │
              ▼           ▼
           t-w          t
           
Timestamps: [t-4, t-3, t-2, t-1, t]
Count = timestamps in [t-w, t]

Smooth limiting, no boundary issues

Pagination

Pagination Methods

Offset Pagination

# Request GET /api/users?offset=20&limit=10 # Response { "data": [...], "pagination": { "offset": 20, "limit": 10, "total": 150, "has_more": true } } # SQL equivalent SELECT * FROM users LIMIT 10 OFFSET 20;

Pros: Easy, jump to any page

Cons: Performance degrades with offset

Cursor Pagination

# First Request GET /api/users?limit=10 # Response { "data": [...], "cursors": { "after": "eyJpZCI6MTB9", "before": "eyJpZCI6MX0=" }, "has_more": true } # Next Page GET /api/users?limit=10&after=eyJpZCI6MTB9 # SQL equivalent SELECT * FROM users WHERE id > 10 ORDER BY id LIMIT 10;

Pros: Consistent, efficient

Cons: Can't jump to arbitrary page

Keyset Pagination

# Request with last seen value GET /api/users?last_id=100&limit=10 # Response { "data": [...], "next_id": 110, "has_more": true } # For sorted data GET /api/users?last_created=2024-01-15T10:00:00Z # SQL SELECT * FROM users WHERE id > 100 ORDER BY id ASC LIMIT 10;

Pros: Best performance for large datasets

Cons: Requires consistent ordering

Link Header Pagination

# Response Headers Link: </api/users?page=1>; rel="first", </api/users?page=5>; rel="prev", </api/users?page=7>; rel="next", </api/users?page=10>; rel="last" # Total count header X-Total-Count: 95

Pagination Comparison

Method Performance Consistency Use Case
Offset ❌ Degrades with offset ❌ May miss/duplicate Small datasets, page UI
Cursor ✅ Constant ✅ Consistent Infinite scroll, APIs
Keyset ✅ Best ✅ Consistent Large datasets, real-time

WebSocket Implementation

Client-Side

// Create WebSocket connection const socket = new WebSocket( 'wss://api.example.com/ws' ); // Connection opened socket.onopen = (event) => { socket.send('Hello Server!'); }; // Listen for messages socket.onmessage = (event) => { console.log('Message:', event.data); }; // Connection closed socket.onclose = (event) => { console.log('Closed'); }; // Send message socket.send(JSON.stringify({ type: 'chat', message: 'Hello' }));

Server-Side (Node.js)

const WebSocket = require('ws'); const wss = new WebSocket.Server({ port: 8080 }); wss.on('connection', (ws) => { console.log('Client connected'); ws.on('message', (message) => { console.log('Received:', message); // Echo back ws.send('Echo: ' + message); }); ws.on('close', () => { console.log('Client disconnected'); }); }); // Broadcast to all function broadcast(message) { wss.clients.forEach((client) => { client.send(message); }); }

Webhook Security

Signature Verification

// Server sends signature const signature = 'sha256=' + crypto.createHmac('sha256', secret) .update(payload) .digest('hex'); // Client verifies function verifyWebhook(payload, sig) { const expected = 'sha256=' + crypto.createHmac('sha256', secret) .update(payload) .digest('hex'); return crypto.timingSafeEqual( Buffer.from(sig), Buffer.from(expected) ); }

Security Best Practices

  • HMAC signature: Verify authenticity
  • HTTPS only: Encrypt in transit
  • Timestamp validation: Prevent replay attacks
  • Idempotency: Handle duplicate deliveries
  • Timeout handling: Respond quickly
  • Retry logic: Exponential backoff

API Versioning

Versioning Strategies

URI Path Versioning

GET /api/v1/users GET /api/v2/users // Most common, explicit

Pros: Clear, cacheable, easy to route

Cons: URL changes, breaking bookmarks

Query Parameter

GET /api/users?version=1 GET /api/users?version=2 // Flexible, optional version

Pros: Optional, backward compatible

Cons: Can be missed, not RESTful

Header Versioning

GET /api/users Accept: application/vnd.api.v1+json X-API-Version: 2 // Clean URLs, discoverable

Pros: Clean URLs, no cache issues

Cons: Harder to test in browser

Host/Subdomain

GET https://v1.api.example.com/users GET https://v2.api.example.com/users // Complete separation

Pros: Complete isolation

Cons: Certificate management

Versioning Best Practices

Key Guidelines

  • Never break existing clients: Use versioning for breaking changes
  • Sunset policy: Announce deprecation timeline
  • Documentation: Maintain versioned docs
  • Default version: Have a fallback version
  • Version headers: Include version in response headers
