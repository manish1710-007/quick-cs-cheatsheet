Complete Reference Guide: Patterns, Communication, Deployment & Production Best Practices for Building Scalable Distributed Systems

⚡ Quick Reference Summary

🎯 Core Principles

Single Responsibility

Each service owns one business capability and does one thing well. Services are loosely coupled but highly cohesive internally.

Independent Deployment

Services can be deployed independently without affecting others. Enables continuous delivery and faster release cycles.

Decentralized Data

Each service manages its own database. No shared databases between services. Data consistency via eventual consistency patterns.

Failure Isolation

Services are designed to fail gracefully. Circuit breakers, bulkheads, and fallbacks prevent cascade failures.

📡 Communication Patterns

Synchronous vs Asynchronous

Synchronous (Request/Response)
REST/gRPC
  • Simple, familiar model
  • Immediate response
  • Tightly coupled in time
  • Cascade failures risk
Asynchronous (Message-Based)
Event-Driven
  • Loose coupling
  • Better fault tolerance
  • Eventual consistency
  • Complexity in debugging

Gateway Responsibilities

🔐 Authentication & Authorization

Centralize auth logic. Validate JWT tokens, check permissions, inject user context into requests before forwarding to services.

// JWT Validation Middleware const validateJWT = async (token) => { const decoded = jwt.verify(token, SECRET); return { userId: decoded.sub, roles: decoded.roles }; };

⚡ Rate Limiting

Protect services from abuse. Implement token bucket or sliding window algorithms per user/IP/API key.

// Rate Limit Config limits: standard: 100 req/min premium: 1000 req/min burst: +20% allowance

🔄 Request/Response Transformation

Transform between formats, aggregate responses, adapt legacy APIs. Enables API versioning without service changes.

📊 Monitoring & Analytics

Centralized logging, metrics collection, request tracing. Single point for observability across all services.

Essential Resilience Patterns

⚡ Retry with Exponential Backoff

Retry transient failures with increasing delays. Add jitter to prevent thundering herd.

Delay: min(baseDelay × 2^attempt, maxDelay) + random(0, jitter) Example: 100ms → 200ms → 400ms → 800ms → give up
Transient Failures

🧱 Bulkhead Pattern

Isolate failures by limiting concurrent calls per service. Prevent one failure from consuming all resources.

// Thread pool per service orderService: max 20 threads paymentService: max 10 threads // If payment fails, order still works
Isolation

🔄 Timeout Pattern

Set aggressive timeouts. Long waits consume resources and indicate problems.

// Timeout recommendations Service call: 2-5 seconds Database query: 1-3 seconds External API: 5-10 seconds Read vs Write: Read shorter
Resource Protection

📉 Fallback Pattern

Provide degraded functionality when service fails. Cache, default values, or queue for later.

Degradation

// Saga Orchestrator Example class OrderSagaOrchestrator { async execute(orderId) { const saga = new Saga(orderId); try { // Step 1: Create Order await this.createOrder(orderId); saga.addCompensation(() => this.cancelOrder(orderId)); // Step 2: Reserve Inventory await this.reserveInventory(orderId); saga.addCompensation(() => this.releaseInventory(orderId)); // Step 3: Process Payment await this.processPayment(orderId); saga.addCompensation(() => this.refundPayment(orderId)); return { success: true }; } catch (error) { // Execute compensating transactions in reverse order await saga.compensate(); return { success: false, error }; } } }

Event Sourcing

📜 Store Events, Not State

Instead of storing current state, store all state-changing events. Current state is derived by replaying events.

// Traditional: Store current state UPDATE orders SET status = 'SHIPPED' WHERE id = 123; // Event Sourcing: Store events INSERT INTO events (type, aggregate_id, data, timestamp) VALUES ('OrderShipped', 123, {"shippedAt": "2024-01-15"}, "2024-01-15T10:30:00Z"); // Event Store events = [ { type: "OrderCreated", data: { items: [...] }, timestamp: "10:00" }, { type: "PaymentProcessed", data: { amount: 150 }, timestamp: "10:05" }, { type: "OrderShipped", data: { trackingId: "ABC" }, timestamp: "10:30" } ] // Reconstruct state by replaying events const rebuildOrder = (events) => { return events.reduce((state, event) => { switch(event.type) { case 'OrderCreated': return { ...state, status: 'CREATED', ...event.data }; case 'PaymentProcessed': return { ...state, status: 'PAID' }; case 'OrderShipped': return { ...state, status: 'SHIPPED', ...event.data }; } }, {}); };
💡 Event Sourcing Benefits:
  • Complete Audit Trail: Every change is recorded with timestamp and context
  • Time Travel: Reconstruct state at any point in time
  • Debugging: Replay events to reproduce issues
  • Event Replay: Rebuild read models when schema changes

Message Patterns

📨 Point-to-Point

One producer, one consumer. Message processed by single consumer. Use for task queues.

📢 Publish-Subscribe

One producer, multiple consumers. Each consumer gets a copy. Use for event broadcasting.

🔄 Request-Reply

Send request, wait for reply on correlation ID. Use for synchronous over async.

⚖️ Competing Consumers

Multiple consumers, one gets each message. Enables horizontal scaling.

Security Best Practices Checklist

🔐 Transport Security

  • mTLS for all service-to-service
  • TLS 1.3 for external APIs
  • Certificate rotation automation
  • No plain-text communication

🎫 Token Security

  • Short-lived access tokens (15min)
  • Refresh token rotation
  • Token revocation mechanism
  • Secure token storage

📝 Input Validation

  • Validate all inputs at API gateway
  • Sanitize for SQL injection
  • Escape for XSS
  • Rate limiting per user/IP

🔒 Secrets Management

  • Vault/Secrets Manager
  • No secrets in code/env files
  • Automatic secret rotation
  • Audit logging for access

⚠️ Anti-Patterns & Best Practices

Common Anti-Patterns

❌ Distributed Monolith

Symptom: Services tightly coupled, deploy together, change together.

Fix: Proper service boundaries, async communication, independent data stores.

❌ Shared Database

Symptom: Multiple services access same database tables.

Fix: Database per service, API for data access, event-driven sync.

❌ Chatty Services

Symptom: Too many fine-grained API calls between services.

Fix: Batch APIs, GraphQL, event-driven aggregation, API composition.

❌ God Service

Symptom: One service does too much, becomes a mini-monolith.

Fix: Split by business capability, apply DDD bounded contexts.

Best Practices Summary

✅ DO:
  • Design for failure - assume everything will fail
  • Keep services small and focused (single responsibility)
  • Use async communication when possible
  • Implement proper observability from day one
  • Automate everything (CI/CD, infrastructure, testing)
  • Version your APIs from the start
  • Implement proper security at every layer
  • Document service contracts and dependencies
⛔ DON'T:
  • Don't start with microservices if you don't need them
  • Don't share databases between services
  • Don't use synchronous calls for everything
  • Don't ignore operational complexity
  • Don't skip automated testing
  • Don't hardcode service locations
  • Don't forget about data consistency
  • Don't create too many services too early

🎯 Quick Decision Guide

When to Choose Microservices? ✅ Choose Microservices when: • Team size > 20 developers • Different services need different scaling • Clear domain boundaries exist • Multiple technology stacks needed • Independent deployment is critical • Services have different reliability requirements ❌ Stay with Monolith when: • Small team (< 10 developers) • Early-stage startup (PMF search) • Simple domain, no clear boundaries • Single deployment unit is acceptable • No scaling challenges yet • Limited operational expertise 🔄 Migrate to Microservices when: • Monolith becomes hard to maintain • Deployment frequency needs to increase • Different parts need different scaling • Team size and complexity grows • Clear bounded contexts emerge

Decision Matrix

Deployment Independence ❌ All or nothing ✅ Per service
Technology Freedom ❌ Single stack ✅ Polyglot
Operational Complexity ✅ Simple ❌ Complex
