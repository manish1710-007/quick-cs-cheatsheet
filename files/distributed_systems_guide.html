📚 Theoretical Foundations

CAP Theorem (Brewer's Theorem)

┌─────────────────────────────────────────────────────────────────────────┐ │ CAP THEOREM │ ├─────────────────────────────────────────────────────────────────────────┤ │ │ │ Consistency (C) │ │ ▲ │ │ /│\ │ │ / │ \ │ │ / │ \ │ │ / │ \ │ │ / │ \ │ │ / │ \ │ │ / │ \ │ │ / │ \ │ │ CA System ◀──┘ │ └──▶ CP System │ │ (Traditional │ (MongoDB, │ │ RDBMS) │ HBase, │ │ │ ZooKeeper) │ │ │ │ │ │ │ │ Availability ◀────────┼──────────▶ Partition │ │ (A) │ Tolerance (P) │ │ │ │ │ │ │ │ AP System ◀──┐ │ └──▶ Must choose C or A │ │ (Cassandra, │ │ when P occurs │ │ DynamoDB, │ │ │ │ CouchDB) │ │ │ │ \ │ / │ │ \ │ / │ │ \ │ / │ │ \ │ / │ │ \ │ / │ │ \ │ / │ │ \ │ / │ │ \│/ │ │ ▼ │ │ Network │ │ Partition │ │ │ │ In reality: P is mandatory in distributed systems │ │ → Real choice is between CP and AP │ └─────────────────────────────────────────────────────────────────────────┘
Consistency (C)

All nodes see the same data at the same time. Every read returns the most recent write or an error.

Linearizable
Availability (A)

Every request receives a response (success or failure), without guarantee it contains the most recent write.

Always Respond
Partition Tolerance (P)

System continues to operate despite arbitrary message loss or partial failure.

Network Reality

FLP Impossibility Result

📐 Fischer, Lynch, Paterson (1985)

Theorem: In an asynchronous distributed system where at least one process may crash, there is no deterministic algorithm that can solve consensus.

Async System + One Faulty Process → No Deterministic Consensus

Implications:

  • Cannot distinguish between a slow process and a crashed one
  • Any consensus algorithm must either use randomness or timeouts
  • Practical systems use partial synchrony assumptions

PACELC Theorem (Extended CAP)

PACELC: Partition (P), Availability (A), Consistency (C), Else (E), Latency (L), Consistency (C) If Partition (P): Choose Availability (A) or Consistency (C) Else (E): Choose Latency (L) or Consistency (C) Examples: - DynamoDB: PA/EL (Available during partition, Low latency normally) - MySQL: PC/EC (Consistent during partition, Consistent normally) - Cassandra: PA/EL (Available during partition, Low latency normally)

Fallacies of Distributed Computing

❌ The Network is Reliable

Networks fail. Cables get cut, routers crash, DNS has issues. Design for network failures with retries, timeouts, and fallbacks.

❌ Latency is Zero

Network latency is real and variable. Light in fiber: ~5ms per 1000km. Add protocol overhead, queuing, processing.

❌ Bandwidth is Infinite

Bandwidth limits exist. Large data transfers compete with other traffic. Consider compression, pagination, streaming.

❌ The Network is Secure

Networks are hostile. Assume attackers exist. Encrypt in transit, authenticate all parties, validate all inputs.

❌ Topology Doesn't Change

Networks are dynamic. Services come and go, load balancers change, routes shift. Use service discovery.

❌ There is One Administrator

Multiple teams, vendors, and systems. Configuration conflicts happen. Document everything, automate configs.

❌ Transport Cost is Zero

Data transfer costs money. Cloud egress fees, CDN costs, inter-region traffic. Optimize data movement.

❌ The Network is Homogeneous

Different protocols, versions, vendors. Interoperability is hard. Use standard protocols, handle versioning.

Vector Clocks for Causal Ordering

// Vector Clock Implementation class VectorClock { constructor(nodeIds) { this.clock = {}; nodeIds.forEach(id => this.clock[id] = 0); } // Increment local clock on event increment(nodeId) { this.clock[nodeId] = (this.clock[nodeId] || 0) + 1; return this; } // Merge clocks on receive merge(otherClock) { for (const [node, time] of Object.entries(otherClock)) { this.clock[node] = Math.max( this.clock[node] || 0, time ); } return this; } // Compare: returns 'before', 'after', 'concurrent', or 'equal' compare(other) { let allLessOrEqual = true; let allGreaterOrEqual = true; let allEqual = true; const allKeys = new Set([ ...Object.keys(this.clock), ...Object.keys(other.clock) ]); for (const key of allKeys) { const a = this.clock[key] || 0; const b = other[key] || 0; if (a > b) allLessOrEqual = false; if (a < b) allGreaterOrEqual = false; if (a !== b) allEqual = false; } if (allEqual) return 'equal'; if (allLessOrEqual) return 'before'; if (allGreaterOrEqual) return 'after'; return 'concurrent'; } } // Usage Example // Node A writes: VC = {A: 1} // Node B receives from A, writes: VC = {A: 1, B: 1} // Node C writes independently: VC = {C: 1} // {A:1, B:1} and {C:1} are CONCURRENT (no causal relationship)
💡 Causal Consistency with Vector Clocks:
  • Each node maintains a vector clock tracking its view of all nodes
  • On local event: increment own entry
  • On send: attach current vector clock to message
  • On receive: merge received clock, increment own entry
  • Concurrent events can be delivered in any order

🤝 Consensus Algorithms

What is Consensus?

Consensus Problem: Multiple nodes need to agree on a single value. Properties: - Agreement: All correct nodes decide the same value - Validity: Decided value was proposed by some node - Termination: All correct nodes eventually decide Use Cases: Leader election, atomic broadcast, distributed transactions, configuration management

Paxos Algorithm

┌─────────────────────────────────────────────────────────────────────────┐ │ PAXOS ALGORITHM FLOW │ ├─────────────────────────────────────────────────────────────────────────┤ │ │ │ ROLES: │ │ - Proposer: Proposes values, coordinates the protocol │ │ - Acceptor: Votes on proposals, stores accepted values │ │ - Learner: Lears the decided value │ │ │ │ PHASE 1: PREPARE │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ Proposer │ │ Acceptor │ │ Acceptor │ │ │ │ │ │ 1 │ │ 2 │ │ │ └────┬─────┘ └────┬─────┘ └────┬─────┘ │ │ │ Prepare(n) │ │ │ │ │───────────────────▶│ │ │ │ │───────────────────────────────────────▶│ │ │ │ │ │ │ │ │ Promise(n, v) │ │ │ │ │◀───────────────────│ │ │ │ │◀───────────────────────────────────────│ │ │ │ │ PHASE 2: ACCEPT │ │ │ │ │ │ │ │ Accept(n, v) │ │ │ │ │───────────────────▶│ │ │ │ │───────────────────────────────────────▶│ │ │ │ │ │ │ │ │ Accepted(n, v) │ │ │ │ │◀───────────────────│ │ │ │ │◀───────────────────────────────────────│ │ │ │ │ │ │ │ │ │ n = proposal number (unique, increasing) │ │ v = proposed value (or highest previously accepted) │ │ │ │ KEY INSIGHT: Majority quorum ensures safety │ │ If any value was chosen, future proposals must use that value │ └─────────────────────────────────────────────────────────────────────────┘
// Paxos Pseudocode - Proposer class Proposer { async propose(value) { // Phase 1: Prepare const proposalNum = this.generateProposalNumber(); const promises = await this.sendPrepare(proposalNum); if (promises.length < this.quorumSize) { throw new Error('No quorum achieved'); } // Use highest previously accepted value (if any) const highestAccepted = this.findHighestAccepted(promises); const proposeValue = highestAccepted ? highestAccepted.value : value; // Phase 2: Accept const accepts = await this.sendAccept(proposalNum, proposeValue); if (accepts.length >= this.quorumSize) { return proposeValue; // Value chosen! } throw new Error('Accept failed, retry with higher proposal number'); } } // Paxos Pseudocode - Acceptor class Acceptor { onPrepare(proposalNum) { if (proposalNum > this.highestPromised) { this.highestPromised = proposalNum; return { promise: true, acceptedValue: this.acceptedValue, acceptedNum: this.acceptedNum }; } return { promise: false }; } onAccept(proposalNum, value) { if (proposalNum >= this.highestPromised) { this.highestPromised = proposalNum; this.acceptedNum = proposalNum; this.acceptedValue = value; return { accepted: true }; } return { accepted: false }; } }

// Raft Leader Election Pseudocode class RaftNode { // Election timeout: 150-300ms random electionTimeout = random(150, 300); heartbeatInterval = 50; startElection() { this.state = 'candidate'; this.currentTerm++; this.votedFor = this.id; this.votesReceived = 1; // Request votes from all peers for (const peer of this.peers) { const response = await peer.requestVote({ term: this.currentTerm, candidateId: this.id, lastLogIndex: this.log.last().index, lastLogTerm: this.log.last().term }); if (response.voteGranted) { this.votesReceived++; } } // Check if won election if (this.votesReceived > this.peers.length / 2) { this.becomeLeader(); } } onRequestVote(request) { // Grant vote if: // 1. Candidate's term >= our term // 2. Haven't voted this term OR already voted for candidate // 3. Candidate's log is at least as up-to-date as ours const logOk = request.lastLogTerm > this.log.last().term || (request.lastLogTerm === this.log.last().term && request.lastLogIndex >= this.log.last().index); if (request.term >= this.currentTerm && (this.votedFor === null || this.votedFor === request.candidateId) && logOk) { this.votedFor = request.candidateId; this.resetElectionTimeout(); return { term: this.currentTerm, voteGranted: true }; } return { term: this.currentTerm, voteGranted: false }; } }

🕐 Time & Clocks

The Problem with Clocks in Distributed Systems

⚠️ Why Physical Clocks Fail:
  • Clock Drift: Hardware clocks drift ~1 second/day
  • Clock Skew: Different nodes have different times
  • NTP Issues: Network delays, clock jumps, backward time
  • Relativity: No global notion of "now" in distributed systems

Logical Clocks (Lamport Timestamps)

// Lamport Clock Implementation class LamportClock { constructor() { this.time = 0; } // Increment on local event increment() { this.time++; return this.time; } // On sending message send() { this.time++; return this.time; } // On receiving message with timestamp receive(receivedTime) { this.time = Math.max(this.time, receivedTime) + 1; return this.time; } } // Usage Example: // Process P1: local event (t=1), send (t=2) // Process P2: receive from P1 (t=max(0,2)+1=3), local event (t=4) // Process P3: local event (t=1), receive from P2 (t=max(1,4)+1=5) // Properties: // - If A → B (A happens before B), then L(A) < L(B) // - But L(A) < L(B) does NOT mean A → B (could be concurrent)

Hybrid Logical Clocks (HLC)

HLC = Physical Time + Logical Counter Structure: (physical_time, logical_counter) Rules: 1. On local event: if pt > physical: (pt, 0), else (physical, counter+1) 2. On send: increment logical counter 3. On receive: physical = max(local_pt, msg_pt, received_physical) if physical changed: counter = max(local_counter, received_counter) else: counter = max(local_counter, received_counter) + 1 Benefits: - Bounded drift from physical time - Monotonically increasing - Captures causality like Lamport clocks - Used in CockroachDB, YugabyteDB

🛡️ Fault Tolerance

Types of Failures

Crash Failure

Node stops responding completely. Detectable via heartbeat/timeouts.

Handleable
Fail-Stop

Node crashes and stays crashed. Detectable and permanent.

Handleable
Omission Failure

Node fails to send/receive messages. Intermittent.

Partially Handleable
Byzantine Failure

Node behaves arbitrarily. May send conflicting messages.

Requires BFT

Fault Tolerance Requirements

Crash Fault Tolerance (CFT): - Tolerates f crash failures with n ≥ 2f + 1 nodes (quorum-based) - Examples: Paxos, Raft, ZooKeeper - Majority quorum ensures at least one correct node Byzantine Fault Tolerance (BFT): - Tolerates f Byzantine failures with n ≥ 3f + 1 nodes - Examples: PBFT, HotStuff, Tendermint - Need 2f+1 matching messages to outvote f malicious nodes Quorum Formula: - Simple majority: n/2 + 1 - Read quorum (R) + Write quorum (W) > N (for consistency) - R = W = majority for strong consistency - R = 1, W = N for highest write consistency - R = N, W = 1 for highest read consistency

Quorum Systems

// Quorum Configuration // N = Replication Factor (total replicas) // W = Write Quorum (nodes to acknowledge write) // R = Read Quorum (nodes to respond to read) // Consistency Level Formulas W + R > N // Strong consistency guarantee W = N // All replicas must respond (highest write consistency) R = N // Read from all replicas (highest read consistency) W + R ≤ N // Eventual consistency possible // Common Configurations for N=3 const configs = { // Strong consistency strongWrite: { W: 3, R: 1 }, // All replicas for write strongRead: { W: 1, R: 3 }, // All replicas for read quorum: { W: 2, R: 2 }, // Majority quorum (most common) // Availability-focused available: { W: 1, R: 1 }, // Any one node (eventual consistency) // Balanced balanced: { W: 2, R: 1 }, // Write majority, read any }; // Write Path async function write(key, value, W) { // Send to all N replicas const responses = await Promise.allSettled( replicas.map(node => node.write(key, value)) ); const acks = responses.filter(r => r.status === 'fulfilled').length; if (acks >= W) { return { success: true, acks }; } throw new Error(`Write quorum not reached: ${acks}/${W}`); } // Read Path with Read Repair async function read(key, R) { const responses = await Promise.allSettled( replicas.map(node => node.read(key)) ); const values = responses .filter(r => r.status === 'fulfilled') .map(r => r.value); if (values.length < R) { throw new Error(`Read quorum not reached`); } // Find most recent version const latest = values.reduce((a, b) => a.timestamp > b.timestamp ? a : b ); // Read repair: update stale replicas values.filter(v => v.timestamp < latest.timestamp) .forEach(v => v.node.repair(key, latest)); return latest.value; }

'commit' : 'abort'; await Promise.all( participants.map(p => p[decision](transactionId)) ); return { success: allReady, transactionId }; } } class Participant { async prepare(transactionId, transaction) { // 1. Acquire locks on resources // 2. Write transaction log (undo/redo) // 3. Return ready (but don't commit yet) try { await this.acquireLocks(transaction); await this.writeLog({ transactionId, transaction, state: 'prepared' }); return { status: 'ready' }; } catch (error) { await this.releaseLocks(transaction); return { status: 'abort', reason: error.message }; } } async commit(transactionId) { // Apply changes permanently, release locks await this.applyTransaction(transactionId); await this.releaseLocks(transactionId); await this.writeLog({ transactionId, state: 'committed' }); } async abort(transactionId) { // Undo any changes, release locks await this.rollback(transactionId); await this.releaseLocks(transactionId); await this.writeLog({ transactionId, state: 'aborted' }); } }

Three-Phase Commit (3PC)

💡 3PC Improvements over 2PC:
  • Adds "CanCommit?" phase before Prepare
  • Non-blocking if coordinator fails (with timeout)
  • Pre-committed state allows safe abort or commit
  • Still vulnerable to network partitions

States: Idle → CanCommit → PreCommit → Commit

⚙️ Essential Distributed Algorithms

Leader Election

Bully Algorithm

Highest ID node becomes leader. Node sends election message to higher-ID nodes. If no response, declares self leader.

O(n²) messages

Ring Algorithm

Nodes arranged in logical ring. Election message circulates with node IDs. Highest ID becomes leader.

O(n) messages

Raft Election

Randomized timeout triggers election. Candidate requests votes, majority wins. Used in etcd, Consul.

Production Ready

Zab Election

Fast leader election with epoch numbers. Highest zxid (transaction ID) wins. Used in ZooKeeper.

Production Ready

Gossip Protocol

┌─────────────────────────────────────────────────────────────────────────┐ │ GOSSIP PROTOCOL │ ├─────────────────────────────────────────────────────────────────────────┤ │ │ │ Epidemic-style information dissemination │ │ │ │ Round 1: Node A knows {A} │ │ ┌─────────┐ │ │ │ A │──────▶ randomly select peer, share state │ │ └─────────┘ │ │ │ │ Round 2: A knows {A,B}, B knows {A,B} │ │ ┌─────────┐ ┌─────────┐ │ │ │ A{AB} │◀───▶│ B{AB} │ │ │ └─────────┘ └─────────┘ │ │ │ │ │ │ ▼ ▼ │ │ ┌─────────┐ ┌─────────┐ │ │ │ C{C} │ │ D{D} │ │ │ └─────────┘ └─────────┘ │ │ │ │ Round 3: All nodes know {A,B,C,D} │ │ │ │ Properties: │ │ - O(log N) rounds to reach all nodes │ │ - Probabilistic eventual consistency │ │ - Robust to node failures │ │ - Scalable (each node contacts O(1) peers per round) │ │ │ │ Used in: Cassandra, DynamoDB, Consul, Kubernetes │ └─────────────────────────────────────────────────────────────────────────┘
// Gossip Protocol Implementation class GossipNode { constructor(id, peers) { this.id = id; this.peers = peers; this.state = new Map(); // Local state this.versions = new Map(); // Version vector this.fanout = 3; // Peers to contact per round } // Periodic gossip (every 1 second) async gossipRound() { // Select random peers (fanout) const targets = this.selectRandomPeers(this.fanout); // Send digest (what versions we have) const digest = Array.from(this.versions.entries()); for (const peer of targets) { const response = await peer.sync(digest); this.mergeState(response.updates); } } // Handle incoming gossip sync(theirDigest) { const updates = []; // Find what they're missing for (const [key, version] of this.versions) { const theirVersion = theirDigest.find(d => d[0] === key)?.[1] || 0; if (version > theirVersion) { updates.push({ key, value: this.state.get(key), version }); } } return { updates }; } mergeState(updates) { for (const { key, value, version } of updates) { if (!this.versions.has(key) || this.versions.get(key) < version) { this.state.set(key, value); this.versions.set(key, version); } } } }

Consistent Hashing

┌─────────────────────────────────────────────────────────────────────────┐ │ CONSISTENT HASHING │ ├─────────────────────────────────────────────────────────────────────────┤ │ │ │ Hash Ring (0 to 2^32-1) │ │ │ │ 0 │ │ │ │ │ Node A ────┼──── Node B │ │ │ │ │ │ │ │ │ │ │ │ Key1 ───┤ │ ├── Key2 │ │ │ │ │ │ │ │ │ │ Node D ────┼──── Node C │ │ │ │ │ 2^32-1 │ │ │ │ Rules: │ │ 1. Hash(node_id) → position on ring │ │ 2. Hash(key) → position on ring │ │ 3. Key belongs to first node clockwise from its position │ │ │ │ With Virtual Nodes (VNodes): │ │ - Each physical node maps to multiple positions │ │ - Better load distribution │ │ - Smoother rebalancing when nodes join/leave │ │ │ │ Node A (150 vnodes) ──▶ 150 positions on ring │ │ Node B (150 vnodes) ──▶ 150 positions on ring │ │ │ │ When Node A fails: Only ~50% of A's keys move (not all from A) │ │ Without vnodes: All of A's keys would go to single node │ │ │ │ Used in: DynamoDB, Cassandra, Riak, Memcached │ └─────────────────────────────────────────────────────────────────────────┘
// Consistent Hashing Implementation class ConsistentHash { constructor(virtualNodes = 150) { this.virtualNodes = virtualNodes; this.ring = new Map(); // position → node this.sortedPositions = []; } hash(key) { // Use MurmurHash3 or similar return murmurhash3(key) % (2 ** 32); } addNode(node) { for (let i = 0; i < this.virtualNodes; i++) { const position = this.hash(`${node}-${i}`); this.ring.set(position, node); } this.sortedPositions = Array.from(this.ring.keys()).sort((a, b) => a - b); } removeNode(node) { for (let i = 0; i < this.virtualNodes; i++) { const position = this.hash(`${node}-${i}`); this.ring.delete(position); } this.sortedPositions = Array.from(this.ring.keys()).sort((a, b) => a - b); } getNode(key) { if (this.sortedPositions.length === 0) return null; const position = this.hash(key); // Find first node clockwise from position let idx = this.sortedPositions.findIndex(p => p >= position); if (idx === -1) { idx = 0; // Wrap around to start } return this.ring.get(this.sortedPositions[idx]); } // Get N nodes for replication getNodes(key, n) { const nodes = []; const position = this.hash(key); let idx = this.sortedPositions.findIndex(p => p >= position); if (idx === -1) idx = 0; while (nodes.length < n && nodes.length < this.sortedPositions.length) { const node = this.ring.get(this.sortedPositions[idx]); if (!nodes.includes(node)) { nodes.push(node); } idx = (idx + 1) % this.sortedPositions.length; } return nodes; } }

🧮 Conflict-Free Replicated Data Types (CRDTs)

What are CRDTs?

💡 CRDT Definition:

Data structures that can be replicated across multiple nodes, where replicas can be updated independently and concurrently, and it's always possible to merge (resolve) any two replicas in a mathematically correct way.

Key Property: Merge operation is commutative, associative, and idempotent.

CRDT Types

Counter (G-Counter)

Grow-only counter. Each node has its own counter. Merge takes max per node.

// G-Counter: Grow-only Counter Node A: [A:5, B:0, C:2] Node B: [A:3, B:4, C:2] Merge: [A:5, B:4, C:2] = 11

PN-Counter

Supports increment and decrement. Two G-Counters: one for increments, one for decrements.

// PN-Counter P (increments): [A:5, B:2] = 7 N (decrements): [A:1, B:0] = 1 Value: P - N = 6

G-Set (Grow-only Set)

Set that only supports add. Merge is union.

// G-Set Node A: {1, 2, 3} Node B: {2, 3, 4} Merge: {1, 2, 3, 4}

OR-Set (Observed-Remove Set)

Supports add and remove. Each element has unique tag. Remove only removes observed tags.

// OR-Set with unique tags Add "x": {x#A1} Add "x": {x#A1, x#B1} // two adds Remove "x" at A: {x#B1} // B's x remains

LWW-Register

Last-Writer-Wins register. Value with timestamp. Merge keeps highest timestamp.

// LWW-Register Node A: {value: "foo", ts: 100} Node B: {value: "bar", ts: 150} Merge: {value: "bar", ts: 150}

LWW-Element-Set

Set with add-set and remove-set. Element is in set if in add-set and not in remove-set (by timestamp).

// LWW-Element-Set Add-set: {x: ts=100, y: ts=150} Remove-set: {x: ts=200} Result: {y} // x removed (200>100)
// CRDT Implementation: G-Counter class GCounter { constructor(nodeId) { this.nodeId = nodeId; this.counts = {}; // nodeId → count } increment(amount = 1) { this.counts[this.nodeId] = (this.counts[this.nodeId] || 0) + amount; return this; } value() { return Object.values(this.counts).reduce((a, b) => a + b, 0); } // Merge: take max of each node's count merge(other) { for (const [node, count] of Object.entries(other.counts)) { this.counts[node] = Math.max( this.counts[node] || 0, count ); } return this; } // Serialize for transmission toJSON() { return { counts: this.counts }; } } // Usage const counterA = new GCounter('A'); const counterB = new GCounter('B'); counterA.increment(5); // A: [A:5], value=5 counterB.increment(3); // B: [B:3], value=3 counterA.merge(counterB); // A: [A:5, B:3], value=8 counterB.merge(counterA); // B: [A:5, B:3], value=8 // Both counters now agree on value 8

⚠️ Anti-Patterns & Common Mistakes

❌ Two-Phase Blocking

Holding locks across service calls while waiting for 2PC. Causes cascading failures.

Fix: Use sagas or eventual consistency.

❌ Distributed Single Thread

Processing requests sequentially across distributed system. Negates scaling benefits.

Fix: Parallelize independent operations.

❌ Ignoring Network Partitions

Assuming network is reliable. No handling for split-brain scenarios.

Fix: Design for partition tolerance, use quorums.

❌ Clock Synchronization Dependency

Relying on synchronized clocks for correctness. NTP can't guarantee sync.

Fix: Use logical clocks or TrueTime-like bounded uncertainty.

❌ Synchronous Everything

All inter-service calls are synchronous. Creates tight coupling, cascade failures.

Fix: Use async messaging where possible.

❌ Over-Engineering Consistency

Strong consistency everywhere when eventual is acceptable.

Fix: Choose consistency level per use case.
