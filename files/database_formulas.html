📐 Entity-Relationship (ER) Model

Basic Concepts

Concept Symbol Description
Entity Rectangle Object with independent existence (e.g., Student, Course)
Attribute Ellipse/Oval Property of an entity (e.g., name, age)
Relationship Diamond Association between entities
Primary Key Underlined Unique identifier for entity
Weak Entity Double Rectangle Depends on owner entity for existence
Identifying Relationship Double Diamond Connects weak entity to owner

Types of Attributes

Attribute Classifications

• Simple: Cannot be divided (e.g., age)
• Composite: Can be divided (e.g., name → first, last)
• Single-valued: One value per entity
• Multi-valued: Multiple values (double ellipse)
• Derived: Computed from other attributes (dashed ellipse)
• Key: Unique identifier (underlined)

Cardinality Constraints

Type Notation Example
One-to-One (1:1) 1 ─── 1 Person ↔ Passport
One-to-Many (1:N) 1 ─── N Department → Employees
Many-to-One (N:1) N ─── 1 Employees → Department
Many-to-Many (M:N) M ─── N Students ↔ Courses
Participation Constraints
• Total Participation: Every entity must participate (double line)
• Partial Participation: Some entities may not participate (single line)

ER to Relational Mapping

Mapping Algorithm

Step 1: For each regular entity → Create relation with all attributes
Step 2: For each weak entity → Create relation with attributes + primary key of owner
Step 3: For each 1:1 relationship → Add foreign key to either side
Step 4: For each 1:N relationship → Add foreign key to N side
Step 5: For each M:N relationship → Create new relation with foreign keys
Step 6: For multi-valued attributes → Create separate relation

Extended ER (EER) Features

Specialization (Top-Down)

Process of defining subclasses from superclass.
• Total: Every superclass entity belongs to subclass
• Partial: Some may not belong to any subclass
• Disjoint: Entity can belong to at most one subclass
• Overlapping: Entity can belong to multiple subclasses

Generalization (Bottom-Up)

Process of defining superclass from subclasses.
Combines common attributes into higher-level entity.

Union (Category)

Represents a collection of entities from different entity types. Symbol: "U" in a circle.

Aggregation

Treats relationship as higher-level entity for participating in other relationships.

📊 Relational Model

Basic Terminology

Relational Term Equivalent Description
Relation Table Collection of tuples (rows)
Attribute Column Named column of a relation
Tuple Row Single record in a relation
Domain Data Type Set of allowable values
Degree/Arity Number of Columns Number of attributes
Cardinality Number of Rows Number of tuples
Schema Structure R(A₁, A₂, ..., Aₙ)

Keys

Super Key
Set of attributes that uniquely identifies a tuple
May contain extra attributes. Any superset of a candidate key.
Candidate Key
Minimal super key (no proper subset is a super key)
A relation can have multiple candidate keys.
Primary Key
Selected candidate key to uniquely identify tuples
NOT NULL, UNIQUE. Only one per relation.
Foreign Key
Attribute(s) referencing primary key of another relation
Enforces referential integrity between tables.
Alternate Key
Candidate keys not chosen as primary key

Relational Algebra Operations

Operation Symbol Notation Description
Selection σ (sigma) σcondition(R) Select rows satisfying condition
Projection π (pi) πattrs(R) Select columns (eliminates duplicates)
Cartesian Product × R × S All combinations of tuples
Union ∪ R ∪ S Tuples in R or S (union compatible)
Intersection ∩ R ∩ S Tuples in both R and S
Difference − R − S Tuples in R but not in S
Join (Theta) ⋈ R ⋈condition S Rows satisfying join condition
Natural Join ⋈ R ⋈ S Join on common attributes
Left Outer Join ⟕ R ⟕ S All R tuples + matching S or NULL
Right Outer Join ⟖ R ⟖ S All S tuples + matching R or NULL
Full Outer Join ⟗ R ⟗ S All tuples from both relations
Division ÷ R ÷ S Tuples that match all S tuples
Rename ρ (rho) ρnewname(R) Rename relation/attributes

Relational Algebra Examples

📌 Example Queries
1. Find all students with GPA > 3.5:
σGPA > 3.5(Student)
2. Get names and majors of all students:
πname, major(Student)
3. Find students enrolled in 'Database' course:
πname(σcname='Database'(Student ⋈ Enrollment ⋈ Course))
4. Find students taking ALL courses:
πsid,cid(Enrollment) ÷ πcid(Course)

Relational Calculus

Tuple Relational Calculus (TRC)

{ t | P(t) }
Query expressed as tuple variable t satisfying predicate P.
Example:
Find students with GPA > 3.5:
{ t | ∃s ∈ Student(s.gpa > 3.5 ∧ t.name = s.name) }

Domain Relational Calculus (DRC)

{ ⟨x₁,x₂,...,xₙ⟩ | P(x₁,x₂,...,xₙ) }
Query uses domain variables (column values) instead of tuple variables.
Example:
{ ⟨n⟩ | ∃g ⟨Student⟩(g > 3.5 ∧ ⟨n,g,_⟩ ∈ Student) }
ℹ️ Equivalence:
Relational Algebra and Relational Calculus are equivalent in expressive power. Both are relationally complete.

📝 SQL (Structured Query Language)

DDL Commands

CREATE TABLE

CREATE TABLE Student (
  sid INT PRIMARY KEY,
  name VARCHAR(50) NOT NULL,
  major VARCHAR(30),
  gpa DECIMAL(3,2) DEFAULT 0.0,
  dept_id INT,
  FOREIGN KEY (dept_id) REFERENCES Department(id)
);

ALTER TABLE

-- Add column
ALTER TABLE Student ADD email VARCHAR(100);

-- Drop column
ALTER TABLE Student DROP COLUMN email;

-- Modify column
ALTER TABLE Student MODIFY name VARCHAR(100);

DROP & TRUNCATE

-- Remove table structure and data
DROP TABLE Student;

-- Remove all data, keep structure
TRUNCATE TABLE Student;

DML Commands

SELECT Statement

SELECT name, major
FROM Student
WHERE gpa > 3.5
ORDER BY name ASC
LIMIT 10;

INSERT Statement

INSERT INTO Student (sid, name, gpa)
VALUES (1, 'John', 3.8);

-- Insert from query
INSERT INTO Student_Backup
SELECT * FROM Student;

UPDATE Statement

UPDATE Student
SET gpa = 4.0, major = 'CS'
WHERE sid = 1;

DELETE Statement

DELETE FROM Student
WHERE gpa < 2.0;

Aggregate Functions

Aggregate Functions
COUNT(*) - count all rows
COUNT(col) - count non-NULL
SUM(col) - sum values
AVG(col) - average
MAX(col) - maximum
MIN(col) - minimum

SELECT major, COUNT(*) AS cnt
FROM Student

Subqueries

Scalar Subquery

-- Single value
SELECT name FROM Student
WHERE gpa = (SELECT MAX(gpa) FROM Student);

IN / NOT IN

SELECT name FROM Student
WHERE major IN ('CS', 'Math', 'Physics');

EXISTS / NOT EXISTS

-- Students enrolled in any course
SELECT name FROM Student s
WHERE EXISTS (
  SELECT * FROM Enrollment e
  WHERE e.sid = s.sid
);

Correlated Subquery

-- Students above avg GPA in their major
SELECT name FROM Student s1
WHERE gpa > (
  SELECT AVG(gpa) FROM Student s2
  WHERE s2.major = s1.major
);

Advanced SQL

Views

CREATE VIEW CS_Students AS
SELECT sid, name, gpa
FROM Student
WHERE major = 'CS';

Indexes

-- Create index
CREATE INDEX idx_name
ON Student(name);

-- Unique index
CREATE UNIQUE INDEX idx_email
ON Student(email);

Window Functions

SELECT name, gpa,
  RANK() OVER (ORDER BY gpa DESC) AS rank,
  ROW_NUMBER() OVER (ORDER BY gpa DESC) AS rn,
  DENSE_RANK() OVER (ORDER BY gpa DESC) AS dr
FROM Student;

CTE (Common Table Expression)

WITH TopStudents AS (
  SELECT name, gpa FROM Student
  WHERE gpa > 3.5
)
SELECT * FROM TopStudents
ORDER BY gpa DESC;

📐 Normalization

Functional Dependencies

Definition
X → Y (X determines Y)
For any two tuples t₁, t₂: if t₁[X] = t₂[X], then t₁[Y] = t₂[Y]

Types of FD

• Trivial: Y ⊆ X (e.g., AB → A)
• Non-trivial: Y ⊈ X
• Completely non-trivial: Y ⊈ X and X ∩ Y = ∅

Armstrong's Axioms

Reflexivity: If Y ⊆ X, then X → Y
Augmentation: If X → Y, then XZ → YZ
Transitivity: If X → Y and Y → Z, then X → Z

Derived Rules

Union
If X → Y and X → Z, then X → YZ
Decomposition
If X → YZ, then X → Y and X → Z
Pseudo-transitivity
If X → Y and YW → Z, then XW → Z

Closure and Keys

Attribute Closure (X⁺)

Initialize: X⁺ = X
Repeat until no change:
For each FD Y → Z in F:
If Y ⊆ X⁺, add Z to X⁺
Return X⁺

Finding Candidate Keys

Find attribute closure of each attribute
Attributes not in any LHS → must be in key
Attributes only on LHS → must be in key
Try combinations until closure = all attributes
📌 Example: Finding Closure
R(A, B, C, D, E) with FDs: A → BC, CD → E, B → D

Find (AB)⁺:
1. (AB)⁺ = {A, B}
2. A → BC: Add B, C → (AB)⁺ = {A, B, C}
3. B → D: Add D → (AB)⁺ = {A, B, C, D}
4. CD → E: Add E → (AB)⁺ = {A, B, C, D, E}
Result: (AB)⁺ = {A, B, C, D, E} → AB is a candidate key

Normal Forms

First Normal Form (1NF)
• All attributes have atomic (indivisible) values
• No repeating groups or arrays
• Each cell contains single value
• Each row is unique
Second Normal Form (2NF)
• Must be in 1NF
• No partial dependency: X → A where X is proper subset of candidate key
• All non-prime attributes fully depend on entire candidate key
Violation:
R(StudID, CourseID, StudName, Grade)
Key: (StudID, CourseID)
FD: StudID → StudName (partial dependency)
Fix: Split into Student(StudID, StudName) and Enrollment(StudID, CourseID, Grade)
Third Normal Form (3NF)
• Must be in 2NF
• No transitive dependency: X → Y → Z (non-prime Z)
• Every non-prime attribute depends ONLY on candidate key
A relation is in 3NF if for every X → A:
  1. X is a super key, OR
  2. A is prime (part of candidate key)
Violation:
R(StudID, Zip, City) with StudID → Zip → City
Fix: Split into Student(StudID, Zip) and ZipCode(Zip, City)
Boyce-Codd Normal Form (BCNF)
• Stronger than 3NF
• Every determinant is a candidate key
A relation is in BCNF if for every X → A:
  X is a super key
Violation:
R(Student, Course, Instructor) with FDs:
(Student, Course) → Instructor
Instructor → Course
Problem: Instructor → Course but Instructor not a key
Fix: Split into R1(Student, Instructor) and R2(Instructor, Course)
Fourth Normal Form (4NF)
• Must be in BCNF
• No multi-valued dependencies (MVD)
MVD X →→ Y: For each X, there is a set of Y values independent of other attributes

Decomposition Properties

Lossless Join Decomposition

R₁ ⋈ R₂ = R (no spurious tuples)
Decomposition is lossless if:
(R₁ ∩ R₂) → R₁ or (R₁ ∩ R₂) → R₂

Dependency Preservation

All FDs in original relation can be enforced by FDs on individual decomposed relations.
Check: (F₁ ∪ F₂ ∪ ... ∪ Fₙ)⁺ = F⁺
Normal Form Eliminates Lossless? Dependency Preserving?
1NF → 2NF Partial dependencies Yes Yes
2NF → 3NF Transitive dependencies Yes Yes
3NF → BCNF Non-key determinants Yes Not always
BCNF → 4NF Multi-valued dependencies Yes Yes

🔄 Transactions

ACID Properties

Atomicity

Transaction is indivisible unit - either all operations complete or none do.
Implementation: Logging, undo on failure

Consistency

Transaction takes database from one consistent state to another.
Implementation: Constraints, triggers

Isolation

Concurrent transactions don't interfere with each other.
Implementation: Locking, MVCC

Durability

Once committed, changes survive system failures.
Implementation: Logging, checkpointing, replication

Transaction States

Active
→
Partially Committed
→
Committed
Active
→
Failed
→

Schedules

Serial Schedule

Transactions execute one after another (no interleaving).
Always consistent but poor performance.

Conflict Serializable

Equivalent to some serial schedule.
No conflicting operations in different order than any serial schedule.
Conflicting Operations
Two operations conflict if:
  1. They belong to different transactions
  2. They access the same data item
  3. At least one is a write operation
Types: Read-Write (RW), Write-Read (WR), Write-Write (WW)

Precedence Graph (Conflict Graph)

Testing Conflict Serializability

Create node for each transaction
Add edge Tᵢ → Tⱼ if Tᵢ executes conflicting operation before Tⱼ
Schedule is conflict serializable ⟺ Graph is acyclic
📌 Example:
Schedule: r₁(A), w₂(A), r₂(B), w₁(B)
Conflicts: r₁(A)-w₂(A), w₁(B)-r₂(B)
Edges: T₁ → T₂ (from A), T₂ → T₁ (from B)
Cycle exists → NOT conflict serializable

Recoverable Schedules

Type Condition
Recoverable If Tᵢ reads from Tⱼ, then Tⱼ commits before Tᵢ
Cascadeless If Tᵢ reads from Tⱼ, Tⱼ must commit before Tᵢ reads
Strict Tᵢ can read/write X only after Tⱼ that wrote X has committed/aborted
💡 Hierarchy:
Strict ⊂ Cascadeless ⊂ Recoverable ⊂ All Schedules

🔐 Concurrency Control

Lock-Based Protocols

Lock Types

• Shared (S): Read lock - multiple transactions can hold
• Exclusive (X): Write lock - only one transaction can hold
SX
S✓✗
X✗✗

Lock Compatibility Matrix

✓ = Compatible (can grant lock)
✗ = Not compatible (must wait)

Two-Phase Locking (2PL)

2PL Protocol
Growing Phase: Transaction can acquire locks but cannot release any
Shrinking Phase: Transaction can release locks but cannot acquire any
✓ Guarantee:
2PL ensures conflict serializability. All schedules under 2PL are conflict serializable.

Variants of 2PL

• Basic 2PL: May cause deadlock
• Strict 2PL: Exclusive locks held until commit/abort
• Rigorous 2PL: All locks held until commit/abort

Lock Point

Point when transaction acquires its final lock.
Transactions can be serialized in order of their lock points.

Deadlock Handling

Deadlock Prevention

Wait-Die: Older waits for younger; younger dies (aborts)
Wound-Wait: Older wounds (aborts) younger; younger waits
Timeout: Abort if waiting too long

Deadlock Detection

Maintain wait-for graph
Periodically check for cycles
If cycle found, abort one transaction (victim selection)

Timestamp-Based Protocols

Timestamp Ordering

TS(Tᵢ) = Unique timestamp for transaction Tᵢ
• W-TS(X): Largest timestamp of transaction that wrote X
• R-TS(X): Largest timestamp of transaction that read X

Rules

Read(X):
  If TS(Tᵢ) < W-TS(X): reject (abort Tᵢ)
  Else: execute, update R-TS(X)
Write(X):
  If TS(Tᵢ) < R-TS(X) or TS(Tᵢ) < W-TS(X): reject
  Else: execute, update W-TS(X)

Multiversion Concurrency Control (MVCC)

MVCC Concept
• Multiple versions of each data item are maintained
• Each write creates a new version with transaction timestamp
• Reads access appropriate version based on timestamp
• No blocking between readers and writers
Used in: PostgreSQL, MySQL (InnoDB), Oracle, MongoDB

Isolation Levels

Level Dirty Read Non-repeatable Read Phantom Read
Read Uncommitted Possible Possible Possible
Read Committed Not Possible Possible Possible
Repeatable Read Not Possible Not Possible Possible
Serializable Not Possible Not Possible Not Possible

Problems Prevented

• Dirty Read: Reading uncommitted data
• Non-repeatable Read: Same query returns different results
• Phantom Read: New rows appear in range queries

SQL Syntax

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

💾 Recovery System

Log-Based Recovery

Log Records

Update log record: ⟨Tᵢ, X, V_old, V_new⟩
Other records:
  ⟨Tᵢ, start⟩ - Transaction started
  ⟨Tᵢ, commit⟩ - Transaction committed
  ⟨Tᵢ, abort⟩ - Transaction aborted
  ⟨checkpoint, L⟩ - Checkpoint with active transaction list L

Undo and Redo

Undo (Rollback)

Restore old value if transaction not committed.
Used for: Active transactions at crash, aborted transactions.
⟨Tᵢ, X, V_old, V_new⟩ → Set X = V_old

Redo (Replay)

Apply changes if transaction committed.
Used for: Committed transactions not written to disk.
⟨Tᵢ, X, V_old, V_new⟩ → Set X = V_new

Recovery Algorithms

UNDO/REDO Algorithm

Find last checkpoint, identify:
Undo list: Active transactions (not committed)
Redo list: Committed transactions after checkpoint
Scan forward from checkpoint, redo all in redo list
Scan backward, undo all in undo list

Write-Ahead Logging (WAL)

Log must be written to stable storage BEFORE data is written to disk.
Rules:
1. Undo rule: Log record written before data page
2. Redo rule: All log records written before commit

Checkpoints

Fuzzy Checkpoint
• Don't stop transactions during checkpoint
• Record list of active transactions
• Background process writes dirty pages
ARIES Recovery
Analysis: Identify dirty pages, active transactions
Redo: Repeat history from last checkpoint
Undo: Roll back uncommitted transactions

🔍 Indexing

Index Types

Type Description Use Case
Primary Index On primary key, ordered file Exact match on key
Clustering Index On non-key, records grouped Range queries on ordering field
Secondary Index On any attribute, dense Non-ordering field lookup
Dense Index Entry for every search key value Fast lookup, more space
Sparse Index Entry for some values only Less space, may need scan

B+ Tree Index

B+ Tree Properties

Order m: each node has ⌈m/2⌉ to m children
• All leaf nodes at same level (balanced)
• Leaf nodes linked for sequential access
• Internal nodes store only keys (pointers)
• Leaf nodes store key-pointer pairs

Complexity

Height h = ⌈log⌈m/2⌉(n)⌉
• Search: O(log n)
• Insert: O(log n) + possible splits
• Delete: O(log n) + possible merges
📌 B+ Tree Example (Order 3)
                [ 30 | 70 ]
               /     |     \
         [10|20] [30|50] [70|80|90]
            |       |        |
         Leaves linked in order

Hash Index

Static Hashing

h(k) = k mod M (M = number of buckets)
• O(1) average search
• Problem: Overflow chains, fixed buckets

Extendible Hashing

• Dynamic directory that grows/shrinks
• Uses prefix of hash value
• Directory size: 2^d where d = global depth

Linear Hashing

• No directory, split buckets incrementally
• Split pointer tracks next bucket to split
• n = current number of buckets, i = ⌈log₂ n⌉
hᵢ(k) = k mod 2ⁱ

Bucket Overflow

Causes:
• Insufficient buckets
• Skewed distribution (many collisions)

Cost Estimation

Cost Components
Total Cost = Disk I/O + CPU Cost + Network Cost
Primary metric: Number of disk block transfers (I/O cost)
Operation Cost (in block transfers)
Table Scan B(R) - all blocks of relation R
Index Scan (point query) h + 1 (h = height of tree)
Index Scan (range query) h + ⌈T/I⌉ (T = tuples, I = index entries per block)
Nested Loop Join B(R) + T(R) × B(S)
Block Nested Loop Join B(R) + ⌈B(R)/(M-1)⌉ × B(S)
Sort-Merge Join Sort(R) + Sort(S) + B(R) + B(S)
Hash Join 3 × (B(R) + B(S))
Notation:
• B(R) = Number of blocks containing R
• T(R) = Number of tuples in R
• M = Number of available memory blocks

Join Algorithms

Nested Loop Join

For each tuple in outer relation R:
For each tuple in inner relation S:
If match, output combined tuple
Cost: O(B(R) × B(S)) worst case

Block Nested Loop Join

Load blocks of R into memory
Scan all of S for matches
Repeat with next blocks of R
Cost: B(R) + B(R)/M × B(S)

Index Nested Loop Join

Cost: B(R) + T(R) × (index lookup cost for S)
Use index on join attribute of S for efficient lookup.

Hash Join

Partition: Hash both relations into buckets
Probe: For each R bucket, match with S bucket
Cost: 3(B(R) + B(S)) typical

Query Optimization

Heuristic Optimization

Push selections down (reduce early)
Push projections down (reduce columns)
Convert cross products to joins
Reorder joins optimally

Cost-Based Optimization

Generate possible query plans
Estimate cost for each plan
Choose plan with minimum cost
Uses statistics: cardinality, selectivity, histograms

Disk Access Time

Total Access Time
Access Time = Seek Time + Rotational Latency + Transfer Time
Seek Time: Move arm to correct track (4-10 ms typical)
Rotational Latency: Wait for sector (~half rotation, 2-4 ms at 7200 RPM)
Transfer Time: Read data (depends on transfer rate)
